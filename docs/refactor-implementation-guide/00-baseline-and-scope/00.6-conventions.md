# 00.6 ŌĆö Conventions

Every per-step file in this guide follows the same structure. If a section is
missing from a step file, that step is not ready to merge.

## Required sections in every step

1. **Goal** ŌĆö what the step achieves in one paragraph.
2. **Report finding** ŌĆö direct quote from the deep-research report.
3. **Pre-state** ŌĆö current code on `main`, with verified line numbers.
4. **Target state** ŌĆö code as it should look after the step lands.
5. **Diff / snippet** ŌĆö paste-ready changes (unified diff or full file).
6. **Testing** ŌĆö exact commands and observations that prove the step works.
7. **Rollback** ŌĆö what to revert, and what data to keep.
8. **Follow-ups** ŌĆö explicit pointers to later steps that depend on this one.

## Snippet conventions

- **Java** snippets compile against `targetSdk 34`.
- **Kotlin** snippets target Kotlin 2.0+ and use `kotlinx-coroutines` 1.9+.
- **Rust** snippets compile against the workspace's pinned Rust toolchain.
- **Slint** snippets use the existing identifiers in `ui/bridge.slint`
  (`Bridge`, `AppState`, `MediaBackendKind`, `PanelBridge`).
- **YAML** snippets use the GitHub Actions schema.
- **Gradle** snippets are Groovy DSL ŌĆö the project's existing DSL choice.

## Diff conventions

- Diffs are **unified diff** format. They can be applied with `patch -p1` on a
  clean `main` checkout, or pasted into a code editor.
- Diff hunks include enough context lines to disambiguate location.
- `(NEW)` annotations mark fully-new files; they have no `---` header.

## Line-number conventions

- Line numbers are **verified on `main`** at the time of writing.
- Line numbers will drift as earlier steps land. The first thing every step
  file says is "re-grep before applying".
- Every step that quotes a line range provides the grep command that finds it.

## "Pre-state on `main`" conventions

- Use `cat`/`sed` to quote the actual file, not paraphrased pseudocode.
- Mark lines that *are* the problem with a `ŌåÉ` arrow comment.
- If the pre-state spans multiple files, list them in dependency order.

## "Testing" conventions

Every step has a table of tests, with three columns:

| Test                   | How                                  | Pre-merge required? |
|------------------------|--------------------------------------|---------------------|
| ŌĆ”                      | exact command or observation         | Yes / No            |

`Yes` means the test must be green before the PR can merge; `No` means it is
useful but optional (e.g., manual verification on a second device).

## "Rollback" conventions

Every step has at least:

- The single-line revert command (`git revert <sha>` or "revert these files").
- A "what state survives the revert" sentence (e.g., "the EncryptedSharedPrefs
  file is left alone").
- An optional partial-rollback path if the full revert is too aggressive.

## Cross-references

- Internal references use relative paths (`../05-ŌĆ”`).
- External references use absolute URLs.
- "See step N" inside prose always pairs with a clickable link.
