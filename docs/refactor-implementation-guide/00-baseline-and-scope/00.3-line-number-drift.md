# 00.3 — Report findings with stale line numbers

The deep-research report cites several specific line ranges that do **not**
match `main`. The architectural claim each one is built on is plausible — but
the cited location is wrong. Re-locate before patching.

## D1 — `MainActivity.java` "lines 2557-2801, 2730-2799"

Report quote:

> "File references: `app/src/main/java/org/fcast/android/sender/MainActivity.java`,
> around lines 2557-2801, 2730-2799."

Actual file size: **1158 LOC.** No line 2557 exists. The relevant blocks are
at:

| What the report meant            | Actual range on `main`           |
|----------------------------------|----------------------------------|
| `onCreate` lifecycle setup       | `325-355`                        |
| `CaptureBroadcastReceiver` setup | `310-323`                        |
| Frame loop / capture path        | `~600-790`                       |

How to re-locate:

```bash
rg -n '^    @Override' app/src/main/java/org/fcast/android/sender/MainActivity.java
rg -n 'CaptureBroadcastReceiver|initializeCapture|cleanupCapture' app/src/main/java/org/fcast/android/sender/MainActivity.java
```

## D2 — `src/lib.rs` "lines 2364-2417, 3216-3345, 3797-3821"

Report quote:

> "Helper functions reconstruct `JavaVM` and `JObject` from raw pointers using
> `unsafe { ...from_raw(...) }`; elsewhere the same boundary also uses `assert!`
> and `unwrap()` when recovering Java VM/activity pointers and when connecting to
> a device. File references: `src/lib.rs`, around lines 2364-2417, 3216-3345,
> 3797-3821."

Actual file size: **3076 LOC.** Only the first range (`2364-2417`) falls inside
the file. Re-locate raw-pointer reconstructions and audit each:

```bash
rg -n 'from_raw|unsafe \{|JavaVM|GlobalRef' src/lib.rs
```

For each hit, decide if it can be replaced with a `GlobalRef`-backed access.
See **step 07** sub-PR 7.7 for the safety-audit playbook.


## Why this matters

When you write a PR that "moves the global BACKEND from line 410 to a typed
registry", a reviewer who checks the original location will fail to find it.
Quote the **verified** line numbers; the report's locations are a translation
artefact, not ground truth.
