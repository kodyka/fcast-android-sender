## Why these numbers matter

| File                                         | Reason it matters                                                                  |
|----------------------------------------------|------------------------------------------------------------------------------------|
| `MainActivity.java` (1158 LOC)               | Largest single Android file. Owns lifecycle, EGL, capture, QR, JNI command parsing.|
| `ScreenCaptureService.java` (84 LOC)         | Small but on the critical-path; sticky service + null deref + LBM.                  |
| `GstPopService.java` (174 LOC)               | Foreground host calling `nativeStart()` on the service main thread.                |
| `MigrationRuntimeService.java` (154 LOC)     | Same shape as GstPopService — refactor in lockstep.                                |
| `src/lib.rs` (3076 LOC)                       | Largest single Rust file. JNI exports + orchestration + UI state callbacks.        |
| `src/backend/persistence.rs` (42 LOC)        | Owns the `gstpop_api_key` plain-JSON storage.                                       |
| `src/backend/mod.rs` (53 LOC)                | Declares the process-global `BACKEND: Lazy<RwLock<…>>`.                            |

## How later steps use these baselines

- Step 03 must shrink `MainActivity.java` by at least ~80 LOC of cleanup additions.
- Step 04 should bring `MainActivity.java` to ≤ 700 LOC.
- Step 07 should bring `src/lib.rs` to ≤ 400 LOC.
- Step 08 should bring `MainActivity.kt` to ≤ 400 LOC.

Treat these as **exit criteria**, not invariants. The point is to demonstrate
that the refactor actually reduced size, not just rearranged it.
