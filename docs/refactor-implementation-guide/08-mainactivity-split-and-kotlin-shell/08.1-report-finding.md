## Why this matters

Java's lack of structured concurrency forces every async path into
`HandlerThread` + callbacks (or worse, plain `Thread`). The current
`MainActivity` mixes three async-coordination styles:

- `HandlerThread` for the OpenGL pump.
- `LocalBroadcastManager` for the projection consent (already removed in
  step 02).
- Inline `runOnUiThread` for projection callbacks.

`kotlinx.coroutines` collapses all three into `lifecycleScope.launch` +
`StateFlow`, which is testable with `kotlinx-coroutines-test`'s
`TestScope`/`runTest`.

## Why "Medium priority"

Steps 02–04 already extract major chunks. Step 08 is the *final* cleanup
that reduces `MainActivity` below 400 LOC and changes the language. It is
not on the critical path for any other step except 10 (Android tests want
coroutines-test infrastructure).

## Constraints

- **JNI symbol names stay stable.** Step 07's symbol-stability guard
  applies here too — Java `MainActivity.nativeFoo()` ↔ Kotlin
  `MainActivity.nativeFoo()` produces the same `Java_..._nativeFoo` symbol.
- **No behavioural change.** This is a language migration + state-holder
  extraction. Performance work goes in step 12.
