## Per-class responsibility table

| Class                          | Owns                                                                              |
|--------------------------------|-----------------------------------------------------------------------------------|
| `MainActivity.kt`              | Slint platform host; calls `AppGraph` to wire dependencies; collects `UiState`.   |
| `QrScannerLauncher.kt`         | ZXing intent integration; parses results and forwards to `RuntimeBridge`.          |
| `SenderController.kt`          | State-holder for backend lifecycle; emits `UiState`; uses `CoroutineScope`.        |
| `UiState.kt`                   | Sealed enum: Disconnected / Starting / Connected / Error.                          |
| `AppGraph.kt`                  | Composition root (step 05).                                                        |
| `RuntimeBridge.kt`             | Backend gateway interface (step 05).                                                |

## What MainActivity is left with

After this step, `MainActivity` is just:

1. `onCreate` → reads `AppGraph`, wires `coordinator` + `controller`,
   collects `controller.uiState`.
2. `onDestroy` → tears down controller + coordinator.
3. JNI `external fun` declarations (for symbol stability — see 08.7).

Total: ≤ 300 LOC.

## Why a separate `shell/` package?

Three reasons:

1. **Discoverability.** The state holder is the second-most-important
   class in the Android shell. Putting it in its own package signals that.
2. **Testability.** Step 10's unit tests target `shell/*`. The package
   becomes the test surface.
3. **Future-proofing.** As the controller grows (more backends, more
   states), additional classes (`SenderEvent.kt`, `SenderEffect.kt`) land
   in the same package.
