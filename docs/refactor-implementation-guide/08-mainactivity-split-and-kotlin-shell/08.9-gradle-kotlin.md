# 08.9 — Gradle / Kotlin setup

## Plugin

```diff
 // app/build.gradle
 plugins {
     id 'com.android.application'
+    id 'org.jetbrains.kotlin.android'
 }
```

Top-level `build.gradle` already has the Android plugin classpath; verify
the Kotlin classpath is present:

```diff
 // build.gradle (top level)
 buildscript {
     dependencies {
         classpath 'com.android.tools.build:gradle:8.5.0'
+        classpath 'org.jetbrains.kotlin:kotlin-gradle-plugin:2.0.20'
     }
 }
```

Use Kotlin 2.0+; earlier versions don't support `data object`.

## JVM target

```diff
 android {
     compileSdk 34
     defaultConfig { … }
+    compileOptions {
+        sourceCompatibility JavaVersion.VERSION_17
+        targetCompatibility JavaVersion.VERSION_17
+    }
+    kotlinOptions {
+        jvmTarget = '17'
+    }
 }
```

JVM target 17 matches Android Gradle Plugin 8.x defaults. If the project
uses AGP 7.x, drop to `'11'`.

## Dependencies

```diff
 dependencies {
+    implementation 'org.jetbrains.kotlin:kotlin-stdlib'
+    implementation 'org.jetbrains.kotlinx:kotlinx-coroutines-android:1.9.0'
+    implementation 'androidx.activity:activity-ktx:1.9.2'
+    implementation 'androidx.lifecycle:lifecycle-runtime-ktx:2.8.4'
+    implementation 'androidx.lifecycle:lifecycle-viewmodel-ktx:2.8.4'

+    testImplementation 'org.jetbrains.kotlinx:kotlinx-coroutines-test:1.9.0'
+    testImplementation 'org.jetbrains.kotlinx:kotlin-stdlib'

     // …existing deps…
 }
```

`kotlinx-coroutines-test` is the vendor-supported test dispatcher. The
report flags it explicitly as the right choice for new controller/state-
holder tests.

## Compile-time toolchain version pin

To prevent Kotlin and AGP version drift:

```diff
 // app/build.gradle
+def kotlin_version = '2.0.20'
+def coroutines_version = '1.9.0'

 dependencies {
-    implementation 'org.jetbrains.kotlin:kotlin-stdlib'
+    implementation "org.jetbrains.kotlin:kotlin-stdlib:$kotlin_version"
-    implementation 'org.jetbrains.kotlinx:kotlinx-coroutines-android:1.9.0'
+    implementation "org.jetbrains.kotlinx:kotlinx-coroutines-android:$coroutines_version"
-    testImplementation 'org.jetbrains.kotlinx:kotlinx-coroutines-test:1.9.0'
+    testImplementation "org.jetbrains.kotlinx:kotlinx-coroutines-test:$coroutines_version"
 }
```

…or use a `libs.versions.toml` if the project already uses Gradle version
catalogues. Either way: pin both Kotlin and coroutines to a known-good
combination.

## Sanity check after the diff applies

```bash
./gradlew --no-daemon :app:dependencies | head -20
./gradlew --no-daemon :app:compileDebugKotlin
./gradlew --no-daemon :app:assembleDebug
```

If `compileDebugKotlin` fails with "JVM target version mismatch", the
`kotlinOptions { jvmTarget }` and `compileOptions { … }` blocks disagree —
fix the `targetCompatibility`.

## Why coroutines 1.9, not 1.8?

`kotlinx-coroutines-test` 1.9 ships `TestScope` with proper
`runTest` semantics; 1.8 had a `runBlockingTest` deprecation that complicates
authoring. Pin to ≥ 1.9.

## Why Kotlin 2.0, not 1.9?

`data object` was stabilised in Kotlin 1.9, but the K2 compiler in 2.0
gives much better incremental compile times for KMP-style multi-module
builds. If the project is already on Kotlin 1.9.20+, this step works
without a Kotlin bump — but step 11 (toolchain upgrade) will bump it
anyway, so combining the bump with step 08 saves a round-trip.

## Single-file conversion check

```bash
./gradlew :app:compileDebugKotlin 2>&1 | tee /tmp/kotlinc.log
grep -c 'error:' /tmp/kotlinc.log
```

Must report `0`. Typical first-conversion errors:

- `Type mismatch: inferred type is X? but Y was expected`
  → add `?` to nullable types or use `requireNotNull(…)`.
- `Smart cast to 'X' is impossible…`
  → assign to a local val first.
- `Function declaration must have a name` (anonymous class issues)
  → use object expression: `object : Listener { ... }`.

## Forgotten plugin in submodule

If a future PR adds a new Kotlin file in `app/`, but Gradle reports
"Kotlin Gradle plugin is not applied", the plugin block at the top of
`app/build.gradle` is missing. Re-run `./gradlew :app:dependencies` —
the output lists which plugins are active.
