# 08.12 — Rollback

## Per-PR revert

```bash
git revert <sha-of-PR-8.X>
```

**Order matters:** if you intend to revert all three, do them in reverse
order (8.3 first, then 8.2, then 8.1). Reverting 8.1 first leaves Kotlin
files in a project that has no Kotlin plugin applied — won't compile.

## Revert PR 8.3 (Kotlin conversion)

Restores `MainActivity.java`. The new `SenderController` and
`QrScannerLauncher` Kotlin classes remain — that's fine, they don't
reference `MainActivity` directly. The Java `MainActivity.java` re-creates
the references to them. Verify:

```bash
rg -n 'SenderController\|QrScannerLauncher' app/src/main/java/
```

Should show the Java side using the Kotlin classes (Kotlin classes are
fully accessible from Java; the reverse is the norm).

## Revert PR 8.2 (controller + UiState)

Removes `shell/SenderController.kt`, `shell/UiState.kt`. The Java
`MainActivity.java` re-introduces the inline backend calls. This works
out of the box because PR 8.2 only *added* the new classes; revert is
just deletion.

## Revert PR 8.1 (Kotlin plugin + QR extraction)

Removes the Kotlin plugin and the dependencies it pulled in. The Java
`MainActivity.java` regains the inline ZXing code.

**Verify** the project still compiles:

```bash
./gradlew --no-daemon :app:assembleDebug
```

If any other Kotlin file remains in `app/` (because steps 05/06 already
landed Kotlin code), the build will fail. In that case:

- Either revert steps 05/06 too (very invasive — not recommended).
- Or keep the Kotlin plugin and only revert the QR extraction. Apply the
  revert as a manual edit (cherry-pick the non-plugin hunks).

## What survives any revert

- The 15 JNI symbol names (the symbol-stability guard from step 07
  catches any drift introduced by the revert PR itself).
- No on-device state.
- No manifest schema change (revert PR 8.3 restores the
  `MainActivity.java` declaration — it was already pointing at the same
  class name, so no manifest edit is needed).

## Long-tail risk: NDK shared library mismatch

`System.loadLibrary("fcastsender")` is in the companion object's `init`
block (`MainActivity.kt` after PR 8.3). If for some reason the revert
moves it to a different location (e.g. a class-level `static {}` in the
Java version), the symbol resolution can race. Workaround:

- Always keep the `loadLibrary` call in *one* place per process. The
  conventional Java location is `static { System.loadLibrary("…"); }`; the
  conventional Kotlin location is `companion object { init { System.loadLibrary("…") } }`.
- Don't duplicate the call across files.

## Partial rollback

If a single Kotlin file is the cause (e.g., `SenderController.kt` has a
race condition), revert just that file:

```bash
git checkout <sha-before-PR-8.2> -- app/src/main/java/org/fcast/android/sender/shell/SenderController.kt
```

…but leave PR 8.1's Kotlin plugin in place. This is the surgical
"keep most of the migration, undo one piece" path.

## Forwards-only fix (preferred for stability bugs)

After PR 8.3 ships, any bug surfaced is *better* fixed forward — fix the
Kotlin code rather than reverting to Java. The reasons:

- Reverting forfeits all of step 10's controller tests.
- Reverting forfeits the Kotlin tooling improvements (lifecycleScope,
  flow operators).
- The fix is usually a one-line change in the new Kotlin file.

Use revert only when the bug is unfixable without significant rework.
