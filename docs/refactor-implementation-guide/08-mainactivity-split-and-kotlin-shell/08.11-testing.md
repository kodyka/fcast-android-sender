## Symbol-diff CI

```bash
./gradlew --no-daemon :app:assembleDebug
nm -D --defined-only app/build/intermediates/cmake/debug/obj/arm64-v8a/libfcastsender.so \
    | grep Java_org_fcast_android_sender \
    | sort > /tmp/symbols.after.txt
diff /tmp/symbols.before.txt /tmp/symbols.after.txt
```

Must be empty. Step 07's symbol-stability gate covers this; step 08
piggy-backs on it.

The conversion can subtly break this if a JNI signature changes (e.g.,
Kotlin `IntArray` versus Java `int[]`). The two map to the same JVM type
`[I`, so the `Java_..._native*` symbol is the same ‚Äî but signatures with
boxed types (`java.lang.Integer`) versus unboxed types (`int`) generate
different symbols. Watch for these.

## Cold-start time methodology

`adb shell am start -W` reports `TotalTime`. Run it 5√ó before the PR, 5√ó
after, take the median:

```bash
for i in 1 2 3 4 5; do
    adb shell am force-stop org.fcast.android.sender
    sleep 1
    adb shell am start -W -n org.fcast.android.sender/.MainActivity | grep TotalTime
done
```

Expected: ‚â§ 1.2√ó the pre-step value (i.e., a 20% regression budget ‚Äî most
of which is the Kotlin stdlib being loaded).

If you see > 1.5√ó cold-start, the lazy initialisations in `AppGraph` are
firing eagerly. Audit `by lazy { ‚Ä¶ }` and switch any premature `lazy` to
genuine lazy (no `init { graph.x }` in `FcastApp.onCreate`).
