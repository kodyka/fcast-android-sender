# 08.10 — Migration recipe (3 sub-PRs)

## PR 8.1 — Add Kotlin support + extract QR

- Apply Kotlin plugin (sub-step 08.9 diff).
- Add Kotlin / coroutines / activity-ktx dependencies.
- Create `QrScannerLauncher.kt`.
- Edit `MainActivity.java`:
  - Add a `QrScannerLauncher qr;` field.
  - Initialise in `onCreate`.
  - Replace the inline ZXing intent integration with `qr.launch()`.
- Net diff: ≤ 200 LOC.

After this PR, `MainActivity.java` shrinks by ~100 LOC. No Kotlin classes
exist yet besides `QrScannerLauncher` and the step-05/06 classes.

## PR 8.2 — Introduce `SenderController` + `UiState`

- Create `UiState.kt`.
- Create `SenderController.kt`.
- Update `MainActivity.java`:
  - Add a `SenderController controller;` field.
  - In `onCreate`, construct it via `AppGraph`.
  - In `onDestroy`, call `controller.shutdown()`.
  - Replace direct `RuntimeBridge` calls with `controller.startBackend(…)`.
  - **Do not yet collect `uiState`** from Java — the activity is still
    Java in this PR. The Slint side already changes state via Rust
    handlers, so the Kotlin flow is *fired-but-unobserved* until PR 8.3.
- Net diff: ≤ 250 LOC.

After this PR, `MainActivity.java` shrinks by ~200 LOC.
`SenderController` is the new state source of truth on the Kotlin side,
but it's only used as a sink (we write to it; we don't read).

Why ship 8.2 separately from 8.3? Two reasons:

- The controller is the most testable piece (step 10). Landing it early
  lets test-writing start in parallel.
- The conversion PR (8.3) is risky enough that having the controller
  already in place reduces the conversion's surface area.

## PR 8.3 — Convert `MainActivity.java` → `MainActivity.kt`

- Use IntelliJ's "Convert Java File to Kotlin" (or Android Studio's).
- Manually clean up:
  - Reorder fields by responsibility.
  - Replace `findViewById<…>` with view binding (if used).
  - Verify `external fun` declarations.
  - Add `@JvmStatic` to companion-object members that JNI / Slint need
    to invoke statically.
- Wire `lifecycleScope.launch { controller.uiState.collect { … } }` in
  `onCreate`.
- Net diff: ≤ 400 LOC. Most of the diff is `-` on the Java side,
  `+` on the Kotlin side; bytecode parity is the gate.

After this PR, `MainActivity.kt` is ≤ 300 LOC.

## Combined recipe (single-PR alternative)

If the team prefers a single PR for the entire step:

- Land PR 8.1, 8.2, 8.3 as commits on one feature branch.
- Squash before merge.
- Resulting PR: ≤ 850 LOC, all reviewable as one unit.

The downside is reverting becomes "all or nothing". The three-PR recipe
is recommended.

## Branch protection considerations

The "convert Java to Kotlin" PR will rename `.java` → `.kt`. Some Git
branch-protection rules treat renames as deletes-+-adds; if so, the diff
shows `-1158 +300` instead of a rename. Configure your reviewer tool
(GitHub uses `?w=1` for whitespace; renames are detected automatically
unless similarity drops below 50%).

## Stop-the-line conditions

Halt the migration if:

- `compileDebugKotlin` cannot resolve the Slint plugin (`slint_compiler.so`
  not loadable from Kotlin classpath — but this should never happen
  because Slint integration is on the Rust side; Kotlin doesn't call Slint
  directly).
- Symbol diff is non-empty after PR 8.3 (means JNI signatures shifted —
  Kotlin's `external fun` doesn't quite match Java's `native` declaration
  for a specific signature; the most common case is `byte[]` vs.
  `java.nio.ByteBuffer`).
- A controller test exercises a scenario that wasn't possible in Java
  (e.g., concurrent `startBackend` + `stopBackend`); fix the controller,
  re-run the test.
