# 08.4 — `QrScannerLauncher.kt`

Wraps ZXing's `IntentIntegrator` so the activity does not deal with the
intent directly.

Create `app/src/main/java/org/fcast/android/sender/qr/QrScannerLauncher.kt`:

```kotlin
package org.fcast.android.sender.qr

import androidx.activity.ComponentActivity
import androidx.activity.result.ActivityResultLauncher
import androidx.activity.result.contract.ActivityResultContracts
import androidx.lifecycle.lifecycleScope
import com.google.zxing.integration.android.IntentIntegrator
import com.google.zxing.integration.android.IntentResult
import kotlinx.coroutines.launch
import org.fcast.android.sender.runtime.RuntimeBridge
import org.json.JSONObject

/**
 * Drives the ZXing scanner and forwards successful scans into the runtime
 * via `RuntimeBridge.graphCommand("qrscan", { …payload… })`.
 *
 * The activity owns the launcher (registered in onCreate) but never sees
 * the result intent — that's parsed inside this class.
 */
class QrScannerLauncher(
    private val activity: ComponentActivity,
    private val runtime: RuntimeBridge,
) {
    private val launcher: ActivityResultLauncher<android.content.Intent> =
        activity.registerForActivityResult(
            ActivityResultContracts.StartActivityForResult()
        ) { result ->
            val parsed: IntentResult? =
                IntentIntegrator.parseActivityResult(result.resultCode, result.data)
            handleResult(parsed)
        }

    fun launch() {
        val integrator = IntentIntegrator(activity)
            .setBeepEnabled(false)
            .setOrientationLocked(true)
            .setDesiredBarcodeFormats(IntentIntegrator.QR_CODE)
            .setPrompt(activity.getString(android.R.string.ok))
        launcher.launch(integrator.createScanIntent())
    }

    private fun handleResult(result: IntentResult?) {
        if (result == null || result.contents.isNullOrEmpty()) return

        val payload = JSONObject().put("payload", result.contents)
        activity.lifecycleScope.launch {
            runtime.graphCommand("qrscan", payload)
        }
    }
}
```

## Dependencies

Add to `app/build.gradle`:

```gradle
dependencies {
    // …existing…
    implementation 'com.journeyapps:zxing-android-embedded:4.3.0'
}
```

If the project already depends on a different ZXing wrapper, keep the
existing one; just route the result through `handleResult` above.

## Why a result-launcher instead of `startActivityForResult`?

`startActivityForResult` is deprecated since API 30. The
`ActivityResultLauncher` lives across configuration changes and integrates
with the activity lifecycle automatically — no manual handling needed.

## How `MainActivity` uses it

```kotlin
private lateinit var qr: QrScannerLauncher

override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    val graph = (application as FcastApp).graph
    qr = QrScannerLauncher(this, graph.runtime)
}

// Later, called from somewhere:
private fun onScanButton() {
    qr.launch()
}
```

The `MainActivity` does not see the result intent.

## Threading

`registerForActivityResult` callback runs on the main thread. The
`lifecycleScope.launch` inside `handleResult` switches to the activity's
default scope (main + supervisor + lifecycle-cancelled). The actual
`runtime.graphCommand` call is `suspend` and switches to `Dispatchers.IO`
internally (see step 05).

## Test impact

`QrScannerLauncher` is hard to unit-test (it depends on the
`ActivityResultLauncher` API which requires an activity). Step 10's
controller tests bypass it — they construct a `SenderController` directly
and call methods on it, without going through QR. If you want a unit test
for the *parsing* logic, factor `handleResult(parsed: IntentResult?)` out
of the class:

```kotlin
internal object QrPayloadParser {
    fun toGraphCommandPayload(result: IntentResult?): JSONObject? {
        if (result == null || result.contents.isNullOrEmpty()) return null
        return JSONObject().put("payload", result.contents)
    }
}
```

…and unit-test `QrPayloadParser`. Not required for this step.
