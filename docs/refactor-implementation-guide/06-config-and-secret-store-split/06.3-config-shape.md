# 06.3 — New JSON config shape

## Before

```json
{
  "gstpop_api_key": "sk-live-abc123…",
  "endpoint": "https://example.com/v1",
  "polling_interval_ms": 5000
}
```

## After

```json
{
  "gstpop_api_key_alias": "gstpop.api_key.v1",
  "endpoint": "https://example.com/v1",
  "polling_interval_ms": 5000
}
```

The secret material lives in the Android Keystore-backed
`EncryptedSharedPreferences`, keyed by the alias string.

## Alias convention

Aliases are namespaced strings:

| Pattern                       | Example                       |
|-------------------------------|-------------------------------|
| `<backend>.<purpose>.v<n>`    | `gstpop.api_key.v1`           |
| `<backend>.<purpose>.v<n>`    | `migration.signing_key.v1`     |

The `v<n>` suffix supports key rotation: when a key is rotated, write
`gstpop.api_key.v2` alongside the old one, then atomically swap the config
JSON to reference `v2`, then delete `v1`. This avoids the "alias points at
no value, app crashes" race.

## Wire format compatibility

`#[serde(default)]` on `gstpop_api_key_alias` means an old config JSON
(without the alias) deserialises to `alias = ""`. The `SecretStore.get("")`
call returns `Err`, which the consumer maps to a clean error. There is no
panic, just a missing-secret error.

The reverse direction — a new config opened by an old binary — is **not**
supported. The migration step (06.8) handles the on-device rewrite once,
synchronously, on app upgrade.
