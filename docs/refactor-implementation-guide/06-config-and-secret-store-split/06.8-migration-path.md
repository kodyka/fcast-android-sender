# 06.8 — On-device migration

When this step ships, existing users have a config JSON with
`gstpop_api_key` (plaintext). The app must, on first run after the upgrade:

1. Read the legacy JSON.
2. Generate an alias (`gstpop.api_key.v1`).
3. Store the key bytes in the SecretStore under that alias.
4. Rewrite the JSON without the secret.
5. Delete any in-memory copies of the plaintext.

Do it synchronously on app start. The legacy file is small; the cost is
trivial.

## Rust-side migration

Create `src/config/migration.rs`:

```rust
//! One-shot migration of legacy `gstpop_api_key` plaintext config blobs into
//! the SecretStore. Idempotent: a config that has already been migrated is
//! left alone.

use std::fs;
use std::path::Path;

use serde::{Deserialize, Serialize};

#[derive(Debug, Deserialize)]
struct LegacyConfig {
    #[serde(default)]
    gstpop_api_key: String,
    #[serde(default)]
    gstpop_api_key_alias: String,
    endpoint: String,
    #[serde(default = "default_polling")]
    polling_interval_ms: u64,
}

#[derive(Debug, Serialize)]
struct ModernConfig {
    gstpop_api_key_alias: String,
    endpoint: String,
    polling_interval_ms: u64,
}

fn default_polling() -> u64 { 5000 }

const DEFAULT_ALIAS: &str = "gstpop.api_key.v1";

pub fn migrate_config_file(path: &Path) -> Result<(), String> {
    let raw = match fs::read_to_string(path) {
        Ok(s)  => s,
        Err(e) => return Err(format!("read_to_string: {e}")),
    };

    let legacy: LegacyConfig =
        serde_json::from_str(&raw).map_err(|e| format!("parse: {e}"))?;

    if !legacy.gstpop_api_key_alias.is_empty() && legacy.gstpop_api_key.is_empty() {
        // Already migrated; nothing to do.
        return Ok(());
    }

    if legacy.gstpop_api_key.is_empty() {
        // No key set; rewrite with empty alias for forwards compat.
        rewrite(path, &legacy, "")?;
        return Ok(());
    }

    // Store the key bytes.
    crate::app::app().secrets()
        .put(DEFAULT_ALIAS, legacy.gstpop_api_key.as_bytes())
        .map_err(|e| format!("put: {e}"))?;

    // Rewrite the file.
    rewrite(path, &legacy, DEFAULT_ALIAS)?;
    Ok(())
}

fn rewrite(path: &Path, legacy: &LegacyConfig, alias: &str) -> Result<(), String> {
    let modern = ModernConfig {
        gstpop_api_key_alias: alias.to_owned(),
        endpoint: legacy.endpoint.clone(),
        polling_interval_ms: legacy.polling_interval_ms,
    };
    let s = serde_json::to_string_pretty(&modern)
        .map_err(|e| format!("serialize: {e}"))?;
    // Best-effort atomic write: write to a tmp file, then rename.
    let tmp = path.with_extension("json.tmp");
    fs::write(&tmp, s).map_err(|e| format!("write tmp: {e}"))?;
    fs::rename(&tmp, path).map_err(|e| format!("rename: {e}"))?;
    Ok(())
}
```

## Call site

Add to `android_main` after `App::init(...)`:

```diff
+    let path = ctx.config_dir().join("migration-config.json");
+    if let Err(e) = crate::config::migration::migrate_config_file(&path) {
+        tracing::warn!("migration config migrate failed: {e}");
+    }
```

The migration runs **once** per app start. It is safe to invoke many times
(idempotent). The `warn!` on failure intentionally does not abort startup —
worst case the user types the API key again.

## Why the `LegacyConfig` struct accepts both fields?

```rust
#[serde(default)]
gstpop_api_key: String,
#[serde(default)]
gstpop_api_key_alias: String,
```

…lets the same struct deserialise pre-migration and post-migration blobs.
The branching in `migrate_config_file` then decides whether work is needed.

## Backup of plaintext during the migration window

Step 06 also expects users to *no longer be in the cleartext window*. There
is no need to also seal a backup copy — the file is overwritten in-place
during migration.

If you want extra paranoia, add a step before the rewrite that writes the
key bytes into the SecretStore *before* deleting them from the JSON. The
function above already does this (see the `put(...)` call before
`rewrite(...)`). The rename is the last step; if the process is killed
between `put` and `rename`, the next start sees both the alias-with-bytes
in the store *and* the cleartext-in-JSON, and the migration retries
successfully.

## Cleanup window

After 2–3 release cycles (or whenever telemetry shows < 1% of users with a
`gstpop_api_key` field), delete the migration code and the legacy field
from `LegacyConfig`. The `gstpop_api_key_alias` becomes the only accepted
field.
