# 06.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings":

> "The migration-runtime config JSON contains a field named
> `gstpop_api_key` whose value is the cleartext API key. The same config is
> written to the on-device config directory and shipped to the Rust runtime
> via JNI. Plaintext secrets in any persisted JSON file are inappropriate;
> the Android Keystore is available, free, and supports
> `MasterKey.Builder().setKeyScheme(MasterKey.KeyScheme.AES256_GCM)`. The
> repo should split the secret material out of the config JSON and resolve
> it at runtime by alias."

Proposed mitigation:

> "**Move secrets out of config JSON.** Introduce a `SecretStore` trait on
> the Rust side and a Kotlin `SecretStore` interface backed by
> EncryptedSharedPreferences. The config JSON keeps a `secret_alias`; the
> runtime calls `app().secrets().get(alias)` to resolve the actual key
> bytes at use time."

## Why "High priority"

Secrets in cleartext are a security regression versus the rest of the app
(which already uses the Android Keystore-backed `EncryptedSharedPreferences`
in at least one other place — see `data/AccountStore.kt` in some forks).
Step 06 closes the gap with a small, well-scoped change.

It is *not* "Highest priority" because the JSON config is currently stored
only in private app storage, which is already inaccessible to other apps on
non-rooted devices. The attack surface is mostly:

- Backup tooling that captures `/data/data/<pkg>/files/...`.
- Logs that accidentally print the JSON.
- A future bug that exposes the config path.

Moving secrets to the Keystore removes all three avenues at once.
