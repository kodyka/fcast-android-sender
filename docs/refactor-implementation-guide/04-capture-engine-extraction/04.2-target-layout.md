# 04.2 — Target layout

After this step:

```text
app/src/main/java/org/fcast/android/sender/
├── MainActivity.java                        (≤ 700 LOC after this step)
└── capture/
    ├── CaptureConfig.kt                      (≤ 30 LOC)
    ├── CapturePermissionResult.kt            (≤ 30 LOC)
    ├── ScreenCaptureCoordinator.kt           (≤ 150 LOC)
    └── CaptureEngine.kt                      (≤ 250 LOC)
```

The directory `app/src/main/java/org/fcast/android/sender/capture/` is new.
Gradle auto-discovers it because it lives under the existing source root.

## Class responsibilities

```
ScreenCaptureCoordinator
├── holds: MediaProjectionManager, MediaProjection
├── exposes: startCapture(activity, captureConfig), stopCapture(), isCapturing
├── listens to: CaptureResultBus (consent result)
├── delegates to: CaptureEngine
└── lifecycle: created in MainActivity.onCreate, shutdown in onDestroy

CaptureEngine
├── holds: EGLDisplay, EGLContext, EGLSurface, FBOs, textures, glHandler/glThread
├── exposes: start(surface, width, height, maxFps), pumpOneFrame(), shutdown()
├── delegates to: native nativeProcessFrame
└── lifecycle: created lazily on first frame, shutdown by Coordinator
```

## Why two classes?

- **Testability.** `CaptureEngine` does not depend on any `Activity` /
  `Context` API; its frame-throttling logic is testable in JVM unit tests via
  the benchmark in step 12.
- **Single-responsibility.** The coordinator deals with consent UX; the
  engine deals with GL bytes. Mixing the two is what made `MainActivity`
  unreviewable.
- **Step-04 boundary.** Each class lands as its own commit; reviewers can
  evaluate one at a time.

## What does **not** move

- `nativeProcessFrame` declaration stays on `MainActivity` for now — JNI
  symbol names track the class declaring the `external fun` / native method.
  Step 08 (Kotlin migration) is the right time to move that declaration too.
- `nativeCaptureStarted`, `nativeCaptureStopped`, `nativeCaptureCancelled` —
  likewise stay on `MainActivity` for now.

The coordinator calls them via a small interface defined in 04.5:

```kotlin
interface CaptureCallbacks {
    fun onCaptureStarted(width: Int, height: Int)
    fun onCaptureStopped()
    fun onCaptureCancelled(reason: String)
    fun onFrame(yPlane: ByteBuffer, uPlane: ByteBuffer, vPlane: ByteBuffer,
                width: Int, height: Int, timestampUs: Long)
}
```

`MainActivity` implements `CaptureCallbacks` by calling its existing native
methods. That keeps the JNI ABI byte-for-byte identical.
