# 04.8 — Migration recipe (3 sub-PRs)

This step is large; land it as three PRs. Each is independently shippable;
the project still builds and casts between PRs.

## PR 04.A — Add new classes, do not wire them yet

Add:

- `app/src/main/java/org/fcast/android/sender/capture/CaptureConfig.kt`
- `app/src/main/java/org/fcast/android/sender/capture/CapturePermissionResult.kt`
- `app/src/main/java/org/fcast/android/sender/capture/ScreenCaptureCoordinator.kt`
- `app/src/main/java/org/fcast/android/sender/capture/CaptureEngine.kt` (skeleton only)

Do **not** edit `MainActivity.java` yet. The new classes compile in isolation;
nothing references them. CI runs the existing capture path unchanged.

`./gradlew :app:assembleDebug` must succeed.

## PR 04.B — Move the projection / coordinator code

Move from `MainActivity` into `ScreenCaptureCoordinator`:

- `MediaProjectionManager.getMediaProjection(...)` calls.
- `MediaProjection.Callback` definition + `registerCallback` / `unregisterCallback`.
- `CaptureResultBus.Listener` handler (was set up in step 02).
- `startActivityForResult` for `createScreenCaptureIntent()`.

In `MainActivity`:

- Add the `coordinator` field.
- Replace the projection-related callsites with `coordinator.startCapture(...)`,
  `coordinator.stopCapture()`, `coordinator.shutdown()`.
- `MainActivity` implements `ScreenCaptureCoordinator.CaptureCallbacks` and
  forwards to the existing `nativeCapture*` methods.

The engine still lives in `MainActivity` at this point — call it from inside
the coordinator via a small back-reference, or temporarily implement the
engine inside the coordinator (later folded into the dedicated file in PR
04.C).

Net diff: roughly +50 / −200 LOC in `MainActivity.java`; new code in
`ScreenCaptureCoordinator.kt`.

## PR 04.C — Move the EGL / frame-loop code into `CaptureEngine`

Move from `MainActivity` into `CaptureEngine`:

- `glThread` / `glHandler` ownership.
- `initEgl` / `setupVirtualDisplay` / `releaseGl`.
- `pumpOneFrame()` body.
- All EGL/GL field declarations.
- `SurfaceTexture.OnFrameAvailableListener` (post to the engine).

Replace `releaseEglResources()` in `MainActivity` with `coordinator.shutdown()`.

Net diff: −300 LOC in `MainActivity.java`; ~250 LOC of new code in
`CaptureEngine.kt`.

## Why three PRs

- **04.A** is unsuitable for revert (zero risk; new code) — easy review.
- **04.B** is the riskiest because it changes the projection-flow callsite.
- **04.C** is high-volume but mechanical (code move with no behavioural
  change) — reviewers byte-diff the original against the move.

A failure during the 04.C diff review can be partially mitigated by skipping
04.C entirely and keeping the engine code inline in the coordinator. The
coordinator file becomes ~400 LOC instead of 150, but the project still
works.

## Cross-step interaction

- Step 03 (lifecycle cleanup) is a hard prerequisite — `coordinator.shutdown()`
  in `onDestroy` only works if `onDestroy` is in place.
- Step 02 (`CaptureResultBus`) is a hard prerequisite — the coordinator's
  `attach()` registers a listener on the bus.
- Step 12 (performance) is a follow-up — it benchmarks the engine and tweaks
  it.
