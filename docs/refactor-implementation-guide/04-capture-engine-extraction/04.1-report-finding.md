# 04.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings":

> "`MainActivity` is responsible for screen capture orchestration, EGL setup,
> Y/U/V framebuffer creation, frame throttling, MediaProjection callback
> handling, native frame processing, and broadcast-receiver registration —
> all in one ~1,100-line class. The capture path updates the external texture,
> performs Y/U/V rendering, reads pixels back from three framebuffers, and
> then hands buffers to native code, while using `Instant.now()` / `Duration.between()`
> for frame throttling."

Proposed mitigation (highest priority):

> "**Extract a `CaptureEngine`.** Pull the MediaProjection setup,
> permission/result flow, EGL/OpenGL setup, framebuffer management, and
> frame-rate throttling into a dedicated class (`CaptureEngine` or
> `ScreenCaptureManager`), unit-test it where possible, and expose a
> minimal callback API to the activity."

This guide splits the responsibility into two classes:

| Class                          | Owns                                                       |
|--------------------------------|------------------------------------------------------------|
| `ScreenCaptureCoordinator`     | Permission flow, projection manager, lifecycle, callbacks. |
| `CaptureEngine`                | EGL/GL state, frame throttling, `nativeProcessFrame`.       |

Two classes instead of one because the engine is unit-testable (frame
throttling, allocation pattern) while the coordinator deals in
`MediaProjectionManager` and `Context` — which only matter on a real device.

## Why "highest priority"

The capture pipeline is the most complex code in the project and currently
has zero isolation. Every later step (Kotlin migration, performance pass,
tests) requires the engine to be a separable unit.
