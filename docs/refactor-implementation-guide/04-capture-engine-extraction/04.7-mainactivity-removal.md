# 04.7 — Strip the extracted code from `MainActivity`

Once `ScreenCaptureCoordinator` + `CaptureEngine` are wired in, delete the
fields and methods that have moved out.

## What to delete

The exact set depends on your branch — re-locate with `rg`:

```bash
rg -n 'MediaProjection|VirtualDisplay|EGL\d|GLES20\.gl(Delete|Read|Bind)|HandlerThread|SurfaceTexture' \
    app/src/main/java/org/fcast/android/sender/MainActivity.java | head -40
```

For each hit, decide:

- Does the call belong inside the coordinator? → move into
  `ScreenCaptureCoordinator.kt` (it should already be there per 04.5).
- Does it belong inside the engine? → move into `CaptureEngine.kt`
  (it should already be there per 04.6).
- Is it dead after the move? → delete from `MainActivity.java`.

## Bridge global wiring

The Slint `start-casting` callback in `src/lib.rs` already forwards to a
`nativeStartCapture`-style helper on `MainActivity`. That call site doesn't
change — only `MainActivity`'s implementation does. Effectively:

```java
// Called from Slint's start-casting callback via JNI:
public void startCastingFromSlint(int scaleW, int scaleH, int maxFps) {
    pendingConfig = new CaptureConfig(scaleW, scaleH, maxFps);
    coordinator.startCapture(pendingConfig);
}
```

If your `MainActivity` does not already have a `startCastingFromSlint` method,
the Slint callback may currently be hand-rolling the consent flow inside
`src/lib.rs`. In that case, route it through `MainActivity` first (one new
`@JvmStatic` helper in MainActivity) and then through the coordinator.
