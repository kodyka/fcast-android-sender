## JVM unit test sketch

This is the test you can actually run on the engine without a device. It only
exercises throttling logic — GL state is mocked out.

```kotlin
package org.fcast.android.sender.capture

import org.junit.Assert.assertEquals
import org.junit.Test

class CaptureEngineThrottlingTest {

    private class TestEngine : CaptureEngine() {
        var frames = 0
        public override fun pumpOneFrame() { frames++ }
    }

    @Test fun throttle_drops_excess_frames() {
        val engine = TestEngine()
        // Pretend we configured 30 fps. minIntervalNanos = 33_333_333.
        val field = CaptureEngine::class.java.getDeclaredField("minIntervalNanos").apply { isAccessible = true }
        field.setLong(engine, 33_333_333L)

        engine.onFrameAvailable()
        engine.onFrameAvailable()
        engine.onFrameAvailable()
        assertEquals(1, engine.frames)
    }
}
```

This test depends on step 10 adding the JUnit + Mockito Gradle deps. If
you're applying step 04 before step 10, add the test as a `pending` file and
keep CI focused on the build/lint matrix.

## Manual leak hunt

```bash
for i in {1..10}; do
    adb shell input tap 100 100   # whatever taps "start"
    sleep 2
    adb shell input tap 100 200   # whatever taps "stop"
    sleep 1
done

adb shell dumpsys meminfo org.fcast.android.sender | head -30
adb shell ps -T -p $(adb shell pidof org.fcast.android.sender) | grep -i capture
```

Expected: `EGL mtrack` flat; the thread listing shows zero `CaptureEngineGL`
threads in the steady state.
