# 04.10 — Rollback

Three sub-PRs (04.A, 04.B, 04.C) — three potential revert points.

## Revert 04.C (riskiest)

```bash
git revert <sha-of-04.C>
```

The coordinator stays; the engine code goes back inside `MainActivity`. The
file gains back ~300 LOC. Capture still works because the coordinator was
calling into the engine — after revert it calls into the inline code via the
same `nativeCapture*` methods.

The engine `.kt` file is left orphaned. Delete it in a follow-up.

## Revert 04.B

```bash
git revert <sha-of-04.B>
git revert <sha-of-04.C>   # required: 04.C depends on 04.B
```

Both coordinator and engine wiring go. New `.kt` files remain orphaned on disk
(no code references them) — safe to leave; clean up in a follow-up.

## Revert 04.A

This is the "remove the new files" revert. Only do it if the new files
themselves caused trouble (e.g., a Kotlin plugin issue blocks the build).
Almost always handled by reverting 04.B + 04.C and leaving 04.A's files
unused.

## What survives any revert

- No on-device state.
- No manifest change.
- No JNI symbol change (the engine calls the same `nativeProcess*` methods).

## Behavioural rollback (no git revert)

If a specific feature regressed but you want to keep the architecture, swap
the engine factory at the coordinator call site:

```kotlin
ScreenCaptureCoordinator(
    applicationContext = appContext,
    callbacks = activityCallbacks,
    engineFactory = { LegacyCaptureEngineAdapter(/* uses the old inline body */) },
)
```

`LegacyCaptureEngineAdapter` is a 40-LOC class that re-implements
`CaptureEngine` against the original inline `MainActivity` code. Useful for
A/B testing a regression hypothesis; do not ship as a permanent path.

## Partial diagnosis tips

If reverting **04.C only** doesn't fix the regression:

- The bug is in the coordinator (04.B) or the consent flow.
- Revert 04.B + 04.C; the project is back to `main`-style architecture.

If reverting **04.C** fixes it:

- The bug is in the engine's frame loop or EGL setup.
- Diff the engine's `pumpOneFrame()` against the original `MainActivity` body
  byte-for-byte. The bug is almost always a transcription error in the move.
