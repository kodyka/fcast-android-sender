# 03.6 — Normalise `captureLock` usage

`captureLock` is a `ReentrantLock`. Two callsites use `synchronized
(captureLock) { … }` (the monitor form), four callsites use `lock()` /
`unlock()`. Pick one — `lock()` / `unlock()` — and apply it consistently.

## Why `lock()` / `unlock()` and not `synchronized`?

`ReentrantLock.lock()` and the `synchronized` block are not interchangeable.
`synchronized` acquires the *intrinsic monitor* of the object, which is
entirely separate from the lock's own state. So today, the two forms are not
protecting each other at all — they are guarding the same field with two
independent mutexes.

The least-surprising fix is to use the explicit lock API everywhere, because
that is also what step 05's controller will use.

## Locate the offending sites

```bash
rg -n 'synchronized\s*\(\s*captureLock' \
    app/src/main/java/org/fcast/android/sender/MainActivity.java
```

Verified hits on `main`:

```
728:            synchronized (captureLock) {
775:            synchronized (captureLock) {
```

## Diff (per site, repeat for each)

### Line ~728

```diff
-            synchronized (captureLock) {
-                // … existing body …
-            }
+            captureLock.lock();
+            try {
+                // … existing body …
+            } finally {
+                captureLock.unlock();
+            }
```

### Line ~775

Same pattern.

## Audit the existing `lock()` callsites

The four existing sites (`captureLock.lock();` at ~633, three `unlock()`s at
~639, ~662, ~751) are *already* in the right form. Verify they pair correctly
with `try { … } finally { unlock(); }`:

```bash
rg -n -B 1 -A 6 'captureLock\.lock' \
    app/src/main/java/org/fcast/android/sender/MainActivity.java
```

If any of them does not have a matching `try { } finally { captureLock.unlock(); }`,
fix it in the same PR. The classic broken shape is:

```java
captureLock.lock();
// body that can throw
captureLock.unlock();
```

…which leaks the lock on any exception. Replace with:

```java
captureLock.lock();
try {
    // body that can throw
} finally {
    captureLock.unlock();
}
```

## Why not a `Lock`-using helper?

A `withLock` Kotlin extension would be the right shape, but the file is still
Java in this step. Step 08 migrates the file to Kotlin and introduces
`captureLock.withLock { … }` at every site in one pass.
