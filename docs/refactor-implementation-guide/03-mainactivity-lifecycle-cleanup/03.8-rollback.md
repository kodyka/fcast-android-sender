# 03.8 — Rollback

## Single-line revert

```bash
git revert <sha-of-the-merged-PR>
```

The PR touches one file: `app/src/main/java/org/fcast/android/sender/MainActivity.java`.

## Why this is safe

Every change in this step is **additive**:

- New `onDestroy` method — adding it cannot regress code paths that don't
  call it. Reverting removes the method.
- New `onStop` method — same logic.
- New `releaseEglResources` method — same logic.
- `captureLock` normalisation — the post-state still acquires/releases the
  same `ReentrantLock`; only the syntactic form changes.

There is no on-device state, no migration, no manifest change.

## What survives the revert

- Nothing on-device.
- The `CaptureResultBus.setListener(null)` line — if step 02's
  `setListener(captureListener)` is still on `main`, reverting step 03 leaves
  the listener un-detached. This is the same shape `main` had before step 03,
  so it is acceptable as a temporary rollback state.

## Partial rollback

If only the `captureLock` normalisation (sub-step 03.6) causes a regression,
revert just that part:

```bash
git checkout <sha-before-03.6> -- app/src/main/java/org/fcast/android/sender/MainActivity.java
```

…and re-apply 03.3, 03.4, 03.5 selectively.

The opposite — keep 03.6 and revert 03.3/03.4/03.5 — is also valid; the lock
normalisation is independent of the lifecycle cleanup.

## How to detect a leak regression after rollback

```bash
adb shell dumpsys meminfo org.fcast.android.sender | head -10
```

A `Native Heap` figure that climbs by ~5 MB per activity restart indicates
the EGL helper is *not* running. If you rolled this step back, that's
expected; consider partial rollback.
