# 03.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings":

> "`MainActivity` registers a `BroadcastReceiver` and a
> `DisplayManager.DisplayListener` and starts a `HandlerThread` named
> "OpenGLThread", but I cannot find matching `unregisterReceiver(...)`,
> `unregisterDisplayListener(...)`, or `quit()` / `quitSafely()` calls. The
> activity also acquires a `MediaProjection.Callback` in a way that is not
> obviously unregistered, and uses a mix of `ReentrantLock` and `synchronized`
> over the same `captureLock`. This is a classic Android leak pattern: each
> Activity instance holds resources that survive `onDestroy()`."

Proposed mitigation (highest priority in the report's table):

> "**Fix `MainActivity` lifecycle.** Add `onDestroy()` (and where appropriate
> `onStop()`) that mirrors `onCreate()` exactly: unregister every receiver
> and listener, stop and join `HandlerThread`s, release the
> `MediaProjection.Callback`, and free EGL/OpenGL resources. Pick one
> synchronisation primitive for `captureLock` and remove the others."

## Why "highest priority"

Activities are routinely destroyed and re-created on configuration changes,
foreground-service notifications, or by the system itself. The current code
leaks one `HandlerThread`, one display listener, one broadcast receiver, and
GL resources per activity instance. The fix is small (≤ 90 LOC), purely
additive, and unlocks safe iteration on every later step that lives inside
`MainActivity`.
