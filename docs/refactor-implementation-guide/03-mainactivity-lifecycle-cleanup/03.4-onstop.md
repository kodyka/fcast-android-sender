# 03.4 — Add `onStop`

`onDestroy` runs only when the activity is actually destroyed. For the common
"user backgrounds the app" case, do the lighter-weight cleanup in `onStop`.

## New method

```java
@Override
protected void onStop() {
    // Stop projection callbacks during background — re-arm in onStart.
    if (mediaProjection != null && mediaProjectionCallback != null) {
        try {
            mediaProjection.unregisterCallback(mediaProjectionCallback);
        } catch (Exception e) {
            Log.w(TAG, "onStop unregisterCallback failed", e);
        }
    }
    super.onStop();
}
```

## Symmetric re-registration in `onStart`

If your `onCreate` (or the capture-start path) installs the projection
callback, hoist that line to a small helper and call it from both places:

```java
private void installMediaProjectionCallback() {
    if (mediaProjection == null) return;
    if (mediaProjectionCallback == null) {
        mediaProjectionCallback = new MediaProjection.Callback() {
            @Override public void onStop() {
                Log.i(TAG, "MediaProjection.Callback#onStop");
                // …existing onStop behaviour…
            }
        };
    }
    mediaProjection.registerCallback(mediaProjectionCallback, glHandler != null ? glHandler : new Handler(getMainLooper()));
}

@Override
protected void onStart() {
    super.onStart();
    installMediaProjectionCallback();
}
```

## When `onStop` is **not** the right place

If capture is supposed to keep running while the app is in the background
(common for screen casting), do **not** unregister the projection callback in
`onStop`. The foreground service keeps the projection alive; the activity
should only release the callback in `onDestroy`.

In that case, this sub-step collapses to:

```diff
+    // onStop intentionally left out — capture must survive background.
+    // See refactor step 03.4 for context.
```

…and the projection callback unregister stays in `onDestroy` (sub-step 03.3).
The deciding factor is whether the foreground-service notification is intended
to keep the projection alive after the user backgrounds the app.

## How to decide

```bash
rg -n 'foregroundServiceType' app/src/main/AndroidManifest.xml
```

If the manifest declares `foregroundServiceType="mediaProjection"` on
`ScreenCaptureService` (it does in some forks), the projection survives
backgrounding; do **not** unregister in `onStop`. On `main` today the manifest
declares `foregroundServiceType="dataSync"` — the activity is the authority,
so unregistering in `onStop` is safe.
