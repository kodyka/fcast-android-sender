# 02.8 — Rollback

## Single-line revert

```bash
git revert <sha-of-the-merged-PR>
```

The PR touches:

- `app/src/main/java/org/fcast/android/sender/ScreenCaptureService.java`
- `app/src/main/java/org/fcast/android/sender/MainActivity.java`
- `app/src/main/java/org/fcast/android/sender/CaptureResultBus.java` (new)
- `app/build.gradle` (+/− `localbroadcastmanager` line)
- `gradle/libs.versions.toml` if used.

## What survives the revert

- `CaptureResultBus.java` becomes unused dead code after the revert. It is
  safe to leave on the filesystem — nothing references it, and reverting the
  rest restores the LBM path. Removing it requires a follow-up commit.

## What does NOT need to be coordinated

- No on-device state. The contract is a one-message channel; nothing
  persists.
- No manifest change. The original LBM path also did not require a `<receiver>`
  declaration (LBM bypasses the manifest).

## Partial rollback

If only the dependency removal causes trouble (e.g., a transitive consumer
appears in CI but not locally):

```diff
- (remove the dependency removal commit)
```

…and keep the code change. The LBM-removal half of the PR continues to work
because none of the *application* code references it; the dependency just
stays on the classpath unused. File a follow-up to investigate the transitive
caller.

## What if `onCaptureResult` regresses?

The new `onCaptureResult(...)` method contains the body of the old
`CaptureBroadcastReceiver.onReceive(...)`. Diff the two by hand against
`main`:

```bash
git show <revert-sha>~1:app/src/main/java/org/fcast/android/sender/MainActivity.java \
    | rg -A 12 'class CaptureBroadcastReceiver' > /tmp/before.txt
rg -A 12 'private void onCaptureResult' \
    app/src/main/java/org/fcast/android/sender/MainActivity.java > /tmp/after.txt
diff /tmp/before.txt /tmp/after.txt
```

If the diff is non-trivial, you transcribed logic incorrectly during the
copy-from-old. Restore the missing pieces and re-test.
