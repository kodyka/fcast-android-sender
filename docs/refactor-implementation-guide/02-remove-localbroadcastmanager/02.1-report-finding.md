# 02.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings":

> "`ScreenCaptureService` uses `LocalBroadcastManager`, which AndroidX
> explicitly deprecated because it behaves like an application-wide event bus
> and encourages layer violations. The library page itself recommends
> migrating to in-process pub/sub primitives (LiveData, reactive streams,
> coroutines)."

Proposed mitigation:

> "**Replace `LocalBroadcastManager` with an in-process channel.** Use a typed
> singleton callback, a `kotlinx.coroutines.flow.SharedFlow`, or an
> `EventBus`-style queue contained to a small, well-tested class. The
> contract is one message: 'projection consent succeeded with (resultCode,
> data)' — there is no reason for a process-wide broadcast surface."

This guide picks the **typed singleton callback** option because:

- It compiles on plain Java (no Kotlin plugin required yet — step 08 brings
  that).
- It is the minimum-machinery change.
- The contract is genuinely one message — using `SharedFlow` would be over-
  engineering for the current scope, and `SharedFlow` arrives naturally with
  step 08's controller introduction.

## Why "highest priority"

The replacement is small, the deprecation is upstream-confirmed, and every
later step that touches `MainActivity` (steps 03, 04, 08) is easier to reason
about once the LBM coupling is gone.
