# 09.1 — Report finding

Two quotes:

> "GitHub Actions builds an arm64 debug APK artifact, while the GitLab
> pipeline still manually assembles an unsigned release APK, which points
> to CI/CD duplication and a release process that is not yet fully
> consolidated."

— `deep-research-report-3.md`, "Testing, CI, risk and rollback".

> "The Android CI setup action downloads SDK, NDK, and GStreamer
> toolchains directly and I did not find caching in that action. That
> means slower builds, duplicate pipeline logic, and a release process
> that still looks transitional."

— same document.

## What "consolidation" means here

- **One pipeline platform.** GitHub Actions is the source of truth.
- **Real release path.** Signed AAB / APK produced by CI, not by a
  developer's laptop.
- **Cached toolchains.** SDK / NDK / GStreamer / Cargo cached on every
  re-run.
- **GitLab kept as fallback.** Frozen but available; deleted only after
  two GitHub-driven releases land successfully.

## Why "Medium priority"

CI/CD is plumbing — it doesn't ship a feature, but it's the *gating*
infrastructure for steps 10 (tests) and 13 (rollout). Without it,
landing those steps requires manual handcrank work.

Estimated CI time savings after caching:

| Stage                            | Cold time (now) | Cached time (target) |
|----------------------------------|------------------|----------------------|
| SDK download                     | ~70 s            | ~5 s                 |
| NDK r25c download                | ~45 s            | ~5 s                 |
| GStreamer 1.28.0 download        | ~50 s            | ~5 s                 |
| Cargo registry                   | ~30 s            | ~5 s                 |
| Cargo target build (incremental) | ~6 min           | ~90 s                |
| **Total per workflow run**       | **~9 min**       | **~3 min**           |

Numbers are estimates based on the observed action durations and the
size of the artifacts; real-world savings will vary.
