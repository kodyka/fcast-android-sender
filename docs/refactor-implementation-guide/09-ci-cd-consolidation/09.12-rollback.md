# 09.12 — Rollback

## Per sub-PR

| Sub-PR                            | Revert effect                                                            |
|-----------------------------------|--------------------------------------------------------------------------|
| 9.A (caching)                     | Toolchains re-download each run; pipelines are slower but green.          |
| 9.B (rename)                      | Workflow filename reverts; update branch-protection checks back.          |
| 9.C (signingConfig)               | Release variant no longer signs from env vars; CI release run fails fast.  |
| 9.D (release workflow + secrets)  | No GitHub release pipeline; releases revert to GitLab manual signing.      |
| 9.E (symbol-stability gate)        | Drift goes unguarded; rely on the step-07 PR's manual check.                |
| 9.F (GitLab freeze)               | Freeze banner removed; the pipeline is "live" again.                        |

## Recommended revert order

If everything goes wrong, revert in reverse:

```bash
git revert <sha-9.F>
git revert <sha-9.E>
git revert <sha-9.D>
git revert <sha-9.C>
git revert <sha-9.B>
git revert <sha-9.A>
```

Then in branch-protection: re-add `Android APK Release` (the old name)
to required checks; remove `Android Debug APK` and `JNI symbol stability
/ diff-symbols`.

## What survives any revert

- The GitLab pipeline (it was never deleted, just frozen with a banner).
- The four GitHub secrets (deleting them is optional; harmless to leave).
- The release keystore (lives outside the repo).
- The composite action (its functionality is unchanged; the cache steps
  were *additive*).

## Rollback the freeze (9.F) but keep the rest

If GitHub Actions release works fine but GitLab is needed temporarily:

1. Remove the banner from `.gitlab-ci.yml` (one-line edit).
2. Verify GitLab pipelines still trigger.

This is the fastest "GitLab is back" path; no force-push, no revert.

## Rollback the release pipeline (9.D) but keep caching (9.A)

If the release pipeline has a bug and you need to roll out via GitLab
again:

1. Comment out the `on: release` trigger in `android-release-apk.yml`.
2. Document that releases go through GitLab "for now".
3. File a follow-up issue to fix `android-release-apk.yml` and re-enable.

This is preferable to a full revert because the caching improvements
from 9.A continue to benefit every PR.

## Worst-case: full re-baseline

If multiple steps regress:

```bash
git checkout main
git checkout -b devin/$(date +%s)-revert-step-9
git revert <oldest-9.A-sha>^..<latest-9.F-sha>  # revert all 9.* PRs at once
git push -u origin devin/$(date +%s)-revert-step-9
gh pr create --title "Revert: refactor step 09 (CI/CD consolidation)" \
    --body "Restoring pre-step-09 CI/CD state."
```

After merging, branch-protection settings need a one-time update.

## Loss-of-signing-keystore scenario

If the release keystore is lost (deleted from password manager, no
backup):

- New version cannot be installed *as an update* on devices with the
  current key. Users must uninstall and reinstall to pick up the new
  signing key.
- The store's package signature changes; Play Store treats it as a new
  app.
- Mitigate by publishing the new app and asking users to migrate.

This is the *only* irrecoverable scenario in step 09 — and even then,
the existing GitLab pipeline doesn't recover the key either. Treat the
keystore as immortal infrastructure; cache 3 copies in 3 places.
