# 09.4 — Add caching to `android-ci-setup`

The composite at `.github/actions/android-ci-setup/action.yml` downloads
SDK command-line tools, NDK r25c, and GStreamer 1.28.0. Add hash-keyed
caches for each. Use `actions/cache@v4` with content-addressed keys.

## Add cache restore steps

Place near the top of `runs.steps:` in
`.github/actions/android-ci-setup/action.yml`:

```yaml
- name: Cache Android SDK command-line tools
  uses: actions/cache@v4
  id: cache-android-sdk
  with:
    path: ${{ runner.tool_cache }}/android-sdk
    key: android-sdk-${{ runner.os }}-cmdline-${{ inputs.sdk-tools-version }}
    restore-keys: |
      android-sdk-${{ runner.os }}-cmdline-

- name: Cache Android NDK r25c
  uses: actions/cache@v4
  id: cache-android-ndk
  with:
    path: ${{ runner.tool_cache }}/android-ndk
    key: android-ndk-${{ runner.os }}-r25c

- name: Cache GStreamer Android 1.28.0
  uses: actions/cache@v4
  id: cache-gstreamer
  with:
    path: ${{ runner.tool_cache }}/gstreamer-android
    key: gstreamer-android-${{ runner.os }}-1.28.0

- name: Cache cargo registry & target
  uses: actions/cache@v4
  with:
    path: |
      ~/.cargo/registry
      ~/.cargo/git
      target
    key: cargo-${{ runner.os }}-${{ inputs.rust-target }}-${{ hashFiles('**/Cargo.lock') }}
    restore-keys: |
      cargo-${{ runner.os }}-${{ inputs.rust-target }}-
```

## Short-circuit downloads on cache hit

For each `curl … && unzip …` step that follows, gate it on the
corresponding `steps.cache-*.outputs.cache-hit`:

```yaml
- name: Download Android NDK r25c
  if: steps.cache-android-ndk.outputs.cache-hit != 'true'
  shell: bash
  run: |
    mkdir -p "${{ runner.tool_cache }}/android-ndk"
    cd "${{ runner.tool_cache }}/android-ndk"
    curl -L -o ndk.zip https://dl.google.com/android/repository/android-ndk-r25c-linux.zip
    unzip -q ndk.zip
    rm ndk.zip

- name: Download GStreamer 1.28.0
  if: steps.cache-gstreamer.outputs.cache-hit != 'true'
  shell: bash
  run: |
    mkdir -p "${{ runner.tool_cache }}/gstreamer-android"
    cd "${{ runner.tool_cache }}/gstreamer-android"
    curl -L -o gst.tar.xz https://gstreamer.freedesktop.org/data/pkg/android/1.28.0/gstreamer-1.0-android-universal-1.28.0.tar.xz
    tar -xf gst.tar.xz
    rm gst.tar.xz
```

## Per-cache key strategy

| Cache                | Key                                                          | Invalidated by                                 |
|----------------------|--------------------------------------------------------------|------------------------------------------------|
| Android SDK          | `android-sdk-${os}-cmdline-${sdk-tools-version}`              | bumping `sdk-tools-version` input.              |
| Android NDK r25c     | `android-ndk-${os}-r25c`                                      | bumping NDK version (e.g. r26d).                |
| GStreamer 1.28.0     | `gstreamer-android-${os}-1.28.0`                              | bumping GStreamer version (covered in step 11).  |
| Cargo                | `cargo-${os}-${rust-target}-${hashFiles('**/Cargo.lock')}`     | any change to `Cargo.lock`.                      |

`Cargo.lock` is the right cache-bust key because:

- Adding a dependency invalidates compilation outputs.
- A version bump invalidates registry contents.
- A target-platform change (e.g. android-x86 added) gets a fresh key.

## Required `inputs:` section update

If the composite doesn't already accept these inputs, add them:

```yaml
inputs:
  rust-target:
    description: "Rust target triple (e.g. aarch64-linux-android)"
    required: true
    default: "aarch64-linux-android"
  sdk-tools-version:
    description: "Android SDK command-line tools version"
    required: false
    default: "11076708"
```

## Optional: maintained CI image

If cache hit-rates are low in practice (PR branches with frequent
`Cargo.lock` churn), the next-best path is a maintained Docker image.
Build it from the existing GitLab Dockerfile:

```yaml
jobs:
  build:
    runs-on: ubuntu-24.04
    container: ghcr.io/kodyka/fcast-android-ci:1.0.0
```

The image bundles SDK + NDK + GStreamer; no per-job download needed. The
trade-off is image size (~2 GB) versus action-cache flakiness.

## Test the cache behaviour

After the PR is merged, re-run the workflow twice:

- First run: cache MISS (cold). Verify all download steps execute.
- Second run: cache HIT. Verify the conditional `if:` skips downloads.

If a HIT happens but the cache is corrupted (extremely rare),
`actions/cache@v4` reports the hit but the restored contents are wrong.
The workaround is to bump the cache key (e.g., add a suffix like `-v2`).
