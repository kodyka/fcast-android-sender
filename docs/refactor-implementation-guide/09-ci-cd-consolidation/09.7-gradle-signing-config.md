# 09.7 — Gradle `signingConfig`

The release workflow (sub-step 09.6) sets four env vars:

- `FCAST_KEYSTORE_PATH` — absolute path to the decoded `.jks`.
- `FCAST_KEYSTORE_PASSWORD` — keystore password.
- `FCAST_KEY_ALIAS` — alias name.
- `FCAST_KEY_PASSWORD` — per-key password.

Wire them into `app/build.gradle`:

```diff
 android {
+    signingConfigs {
+        release {
+            storeFile     file(System.getenv("FCAST_KEYSTORE_PATH") ?: "release.jks")
+            storePassword System.getenv("FCAST_KEYSTORE_PASSWORD") ?: ""
+            keyAlias      System.getenv("FCAST_KEY_ALIAS")         ?: ""
+            keyPassword   System.getenv("FCAST_KEY_PASSWORD")      ?: ""
+        }
+    }
     buildTypes {
         release {
+            signingConfig signingConfigs.release
             minifyEnabled false
             proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
         }
     }
 }
```

## What this does NOT do

- Does not enable `minifyEnabled`. Performance pass (step 12) might turn
  it on; not in scope for step 09.
- Does not enable `shrinkResources`. Same reasoning.
- Does not enable v1 signing scheme (jarsigner). `assembleRelease`
  produces v2+v3 by default.

## Local development override

Local developers without a keystore should still be able to build the
release variant (e.g. for `proguard` testing). The fallback
`?: "release.jks"` causes a friendly Gradle error if no env var is set:

```
> Task :app:packageRelease FAILED
…
Caused by: java.io.FileNotFoundException: release.jks (No such file or directory)
```

…which is clearer than a `NullPointerException`. Document this in
`CONTRIBUTING.md` (step 13's release checklist references it).

## Version-name / version-code injection

The workflow passes `-PversionName=$FCAST_VERSION_NAME` and
`-PversionCode=$FCAST_VERSION_CODE` to `assembleRelease`. The Gradle file
already consumes these via `project.properties`:

```bash
rg -n 'versionName\|versionCode' app/build.gradle
```

If the file doesn't already use the project properties, add:

```diff
 android {
     defaultConfig {
-        versionCode 1
-        versionName "0.1.0"
+        versionCode  (project.findProperty('versionCode')?.toInteger()    ?: 1)
+        versionName  (project.findProperty('versionName')?.toString()     ?: '0.1.0')
     }
 }
```

This makes the version override-able from CI without editing
`build.gradle`.

## Verify locally

```bash
cd /path/to/repo
export FCAST_KEYSTORE_PATH=/path/to/dev-release.jks
export FCAST_KEYSTORE_PASSWORD=changeit
export FCAST_KEY_ALIAS=fcast
export FCAST_KEY_PASSWORD=changeit
./gradlew --no-daemon :app:assembleRelease \
    -PversionName=0.0.0-dev \
    -PversionCode=1
$ANDROID_HOME/build-tools/34.0.0/apksigner verify --verbose \
    app/build/outputs/apk/release/app-release.apk
```

`apksigner verify --verbose` should print "Verified using v2 scheme (APK
Signature Scheme v2): true".

If the keystore was generated with `keytool -genkey -keyalg RSA`, all
schemes (v1, v2, v3) are supported automatically. The pipeline uses v2+v3
by default; v1 is unnecessary on API 24+.

## Generate a dev keystore

For local testing:

```bash
keytool -genkey -v \
    -keystore dev-release.jks \
    -keyalg RSA -keysize 2048 \
    -validity 36500 \
    -alias fcast \
    -storepass changeit -keypass changeit \
    -dname "CN=Fcast Dev, OU=Dev, O=fcast, L=local, S=local, C=US"
```

Do **not** commit this file. The release keystore is provisioned via
GitHub secret (sub-step 09.9).

## Why fail-fast instead of silent fallback to a debug key?

If the workflow silently signs with a debug key when the secret is
missing, the resulting APK can't be installed over a previously-installed
release-signed APK ("INSTALL_FAILED_UPDATE_INCOMPATIBLE"). The fail-fast
behaviour ("FCAST_KEYSTORE_BASE64 secret is missing") in 09.6 is the
right call.
