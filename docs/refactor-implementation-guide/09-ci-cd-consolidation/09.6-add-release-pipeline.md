# 09.6 — Add `android-release-apk.yml` (signed release)

Full file. Place at `.github/workflows/android-release-apk.yml`:

```yaml
name: Android Release APK

on:
  release:
    types: [published]
  workflow_dispatch:
    inputs:
      version-name:
        description: "Version name (e.g. 1.4.0)"
        required: true
      version-code:
        description: "Version code (integer)"
        required: true

permissions:
  contents: write

jobs:
  build-release:
    runs-on: ubuntu-24.04
    env:
      CARGO_TERM_COLOR: always
    steps:
      - uses: actions/checkout@v4

      - name: Set up Android CI toolchain
        uses: ./.github/actions/android-ci-setup
        with:
          rust-target: aarch64-linux-android

      - name: Build GStreamer Android JNI glue
        run: bash ci/build-gstreamer-android-glue.sh

      - name: Build arm64 Rust Android library (release)
        run: bash ci/build-rust-android-lib.sh release

      - name: Decode signing keystore
        env:
          KEYSTORE_BASE64: ${{ secrets.FCAST_KEYSTORE_BASE64 }}
        run: |
          if [[ -z "${KEYSTORE_BASE64}" ]]; then
            echo "::error::FCAST_KEYSTORE_BASE64 secret is missing"; exit 1
          fi
          echo "$KEYSTORE_BASE64" | base64 -d > $RUNNER_TEMP/release.jks
          test -s "$RUNNER_TEMP/release.jks" || { echo "::error::decoded keystore is empty"; exit 1; }

      - name: Build signed release APK
        env:
          FCAST_KEYSTORE_PATH:     ${{ runner.temp }}/release.jks
          FCAST_KEYSTORE_PASSWORD: ${{ secrets.FCAST_KEYSTORE_PASSWORD }}
          FCAST_KEY_ALIAS:         ${{ secrets.FCAST_KEY_ALIAS }}
          FCAST_KEY_PASSWORD:      ${{ secrets.FCAST_KEY_PASSWORD }}
          FCAST_VERSION_NAME:      ${{ inputs.version-name || github.event.release.tag_name }}
          FCAST_VERSION_CODE:      ${{ inputs.version-code }}
        run: |
          ./gradlew --no-daemon assembleRelease \
            -PversionName=$FCAST_VERSION_NAME \
            -PversionCode=$FCAST_VERSION_CODE

      - name: Verify APK signature
        run: |
          $ANDROID_HOME/build-tools/34.0.0/apksigner verify --verbose \
            app/build/outputs/apk/release/app-release.apk

      - name: Upload signed APK as workflow artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-release-${{ inputs.version-name || github.event.release.tag_name }}
          path: app/build/outputs/apk/release/app-release.apk
          retention-days: 30

      - name: Attach signed APK to GitHub release
        if: github.event_name == 'release'
        uses: softprops/action-gh-release@v2
        with:
          files: app/build/outputs/apk/release/app-release.apk

      - name: Compute APK sha256
        if: github.event_name == 'release'
        run: |
          sha256sum app/build/outputs/apk/release/app-release.apk \
            | tee app-release.sha256

      - name: Attach sha256 to release
        if: github.event_name == 'release'
        uses: softprops/action-gh-release@v2
        with:
          files: app-release.sha256
```

## What this workflow does NOT do

- Does not push to Play Store. Play Store upload is intentionally manual
  for now — add an `action-publish` step in a follow-up PR if/when the
  team wants automation.
- Does not produce an AAB (`bundleRelease`). Add it as a parallel step if
  Play Store internal-app-sharing is needed.
- Does not sign the JNI library separately. The APK signing covers the
  embedded `.so`.

## Required Gradle change

The `signingConfigs.release` block in `app/build.gradle` is the only
project-side prerequisite — see sub-step 09.7.

## Why `actions/checkout@v4`?

v4 supports shallow-fetch-by-depth and faster clone times on
`runs-on: ubuntu-24.04`. Pin to v4 to avoid the v3 default (which is
slower on large checkouts).

## Why `softprops/action-gh-release@v2`?

It's the de-facto standard. Alternatives like `ncipollo/release-action`
also work, but `softprops` has the cleanest API for "attach file to
existing release".

## Secrets the workflow needs

See sub-step 09.9. The workflow won't even start if `KEYSTORE_BASE64` is
missing (the explicit guard in the "Decode signing keystore" step fails
fast).

## Workflow_dispatch UX

When triggering manually:

1. Actions tab → "Android Release APK" → "Run workflow".
2. Fill in `version-name` (e.g. `1.4.0`).
3. Fill in `version-code` (e.g. `14`).
4. Submit.

A signed APK is attached as an artifact (no GitHub release object
involved). Use this for pre-release smoke testing.

## Release-triggered UX

When triggered by a GitHub release:

1. Create a release tag (e.g. `v1.4.0`).
2. Publish the release.
3. The workflow runs automatically.
4. The signed APK is attached to the release page.

This is the "production" path the GitLab pipeline currently fills.

## Compatibility with PR 8.3's Kotlin classes

`assembleRelease` triggers the full Gradle build. After step 08, this
also compiles the new Kotlin code. No additional workflow tweak required
— the Kotlin plugin auto-engages when `.kt` files are present.
