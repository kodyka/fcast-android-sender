# 09.9 — Required GitHub secrets

## Secrets to provision

| Secret                         | Source                                                                 |
|--------------------------------|------------------------------------------------------------------------|
| `FCAST_KEYSTORE_BASE64`        | `base64 -w0 release.jks`. ~3–4 KB after base64.                         |
| `FCAST_KEYSTORE_PASSWORD`      | Keystore password.                                                      |
| `FCAST_KEY_ALIAS`              | Alias name in the keystore (e.g. `fcast`).                              |
| `FCAST_KEY_PASSWORD`           | Per-key password.                                                       |

## How to set

```bash
# Sign in once
gh auth login --hostname github.com --git-protocol https --web

# From the repo root
gh secret set FCAST_KEYSTORE_BASE64 < <(base64 -w0 ./release.jks)
gh secret set FCAST_KEYSTORE_PASSWORD --body 'YOUR_KEYSTORE_PASSWORD_HERE'
gh secret set FCAST_KEY_ALIAS         --body 'fcast'
gh secret set FCAST_KEY_PASSWORD      --body 'YOUR_KEY_PASSWORD_HERE'

# Verify
gh secret list
```

## Why base64 the keystore?

GitHub secrets are designed for *strings*. A binary `.jks` file can't be
pasted into the secret UI directly. Base64 (with `-w0` to disable line
wrapping) round-trips through GitHub safely.

The workflow's "Decode signing keystore" step (sub-step 09.6) reverses
the encoding back to the binary file.

## Scope: repository or environment?

If the team uses GitHub Environments (e.g., a `production` environment
that requires manual approval), put the secrets there instead of at the
repo level:

1. Settings → Environments → New environment "production".
2. Required reviewers: list the release approvers.
3. Add the four secrets at the environment level.
4. In `android-release-apk.yml`, add `environment: production` to the
   `build-release` job.

This adds a manual-approval gate before the release job runs — useful
when the keystore is sensitive.

For a smaller team, repo-level secrets are fine.

## Backup the keystore

The keystore is **non-recoverable**: lose it, and you lose the ability to
ship updates to the same app ID. Store a copy in:

- A 1Password / Bitwarden / hardware-secured location.
- An encrypted offline backup.

Do **not** commit the keystore to the repo, even via `git-crypt`. The
encryption overhead doesn't pay off for a single artifact that needs
out-of-band access for emergency releases.

## Rotation policy

If the keystore is suspected compromised:

1. Cannot rotate — Android verifies APK signatures against the original
   key. A new keystore means a new app on Play Store (different package
   signature).
2. The mitigation is "publish a security update from the same key and
   accelerate device pickup".

This is why the keystore matters more than any other secret.

## Compromise protocol

If `FCAST_KEYSTORE_BASE64` leaks:

1. The leaked key cannot sign new versions on a fresh install — only
   updates to existing installs.
2. Generate a new keystore for the *next major version* (which can be
   published as a new Play Store app with a different package name).
3. Communicate to users that the new version requires a fresh install.
4. Optionally, push a final v-of-old-key update that tells users to
   migrate.

This is a worst-case scenario; the secret hygiene in 09.6 (fail-fast on
missing, no `echo $KEYSTORE_BASE64`) is designed to minimise the chance.

## Pre-merge checklist for PR 9.D (release workflow)

- [ ] All four secrets set via `gh secret list`.
- [ ] Local dry-run of the workflow via `workflow_dispatch` with a
       `0.0.0-test` version-name.
- [ ] Resulting artifact installs on a clean device via
       `adb install -r app-release.apk`.
- [ ] `apksigner verify --verbose` reports "Verified using v2 / v3".
