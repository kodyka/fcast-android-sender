# 09.8 — Freeze GitLab pipeline (do not delete)

Add a banner comment to `.gitlab-ci.yml`; keep `when: manual`; do **not**
delete the file.

## Banner diff

```diff
+# DEPRECATED: kept for rollback until two consecutive GitHub-driven releases
+# ship successfully. Do not extend.
+#
+# See docs/refactor-implementation-guide/09-ci-cd-consolidation/.
+#
+# Manual trigger is preserved so the team retains an emergency build path
+# in case the GitHub release workflow regresses.
+
 buildAndroidSenderDockerContainer:
   stage: build-container
   …
```

## Per the report

> "Keep GitLab pipeline frozen but available until two successful GitHub
> release runs."

— `deep-research-report-3.md`, rollback table.

## What "frozen" means

- No new GitLab pipeline edits (no new stages, no new jobs, no new
  variables).
- The existing `when: manual` triggers stay in place.
- The build still produces the same `app-release-unsigned.apk` artifact.
- The pipeline is **runnable** in an emergency.

## What "frozen" does NOT mean

- Does not mean "deleted". The file remains in version control.
- Does not mean "disabled". GitLab UI still shows the pipeline.
- Does not mean "broken". The pipeline must still successfully run if
  manually triggered.

## When to actually delete

Per the report's rollback table:

> "Two successful GitHub release runs."

After GitHub-driven releases v1 and v2 ship cleanly, open a PR that:

1. Deletes `.gitlab-ci.yml`.
2. Removes the GitLab CI badge from `README.md`.
3. Disables the GitLab CI integration in repo settings.
4. Updates `docs/refactor-implementation-guide/13-rollout-and-rollback/`
   to mark this as a completed milestone.

Schedule that PR for "after release v2 is in the wild for ≥ 1 week
without rollback".

## Communication

Send a one-line note to the team channel when the freeze lands:

> "GitLab pipeline is now frozen (manual-only fallback). Production
> releases go through GitHub Actions starting today. Frozen state lasts
> until two GH releases ship cleanly."

…and update any internal runbook that mentions the GitLab pipeline as
the release path.

## Smoke test the frozen pipeline

After landing the banner:

1. Navigate to GitLab → CI/CD → Pipelines.
2. Manually trigger the `assembleRelease` job.
3. Verify `app-release-unsigned.apk` is still produced.
4. Cancel any other auto-triggered jobs (the freeze does not remove
   triggers, just signals "do not extend").

## Rollback the freeze

If GitHub Actions is broken and the team needs to revert:

```bash
git revert <sha-of-freeze-PR>
```

Or, faster, manually edit `.gitlab-ci.yml` and delete the banner. Either
way the pipeline keeps working — the banner is *documentation*, not a
functional change.
