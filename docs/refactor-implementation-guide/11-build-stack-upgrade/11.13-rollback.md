# 11.13 — Rollback

Each sub-PR is independently revertable. The riskiest is **PR 11.4** (AGP
9). If a regression appears only after 11.4, revert that PR and stay on
AGP 8.13 + Gradle 9 — that combination is supported and gives you most
of the build-perf wins without the AGP 9 risks.

## Recommended revert order

If everything goes wrong, revert in reverse:

```bash
git revert <sha-11.7>
git revert <sha-11.6>
git revert <sha-11.5>
git revert <sha-11.4>
git revert <sha-11.3>
git revert <sha-11.2>
git revert <sha-11.1>
```

Then in CI: clear the SDK/NDK/Gradle caches once (sub-step 09.4) to
force fresh downloads at the old versions.

## Partial revert: keep Gradle 9, drop AGP 9

The "stop at 8.13" exit door:

```bash
git revert <sha-11.7>     # NDK back to r25c
git revert <sha-11.4>     # AGP back to 8.13
```

Leave PRs 11.1, 11.2, 11.3 (Gradle 9, AGP 8.13, compileSdk 35) in place.
This is the **recommended fallback** if AGP 9 surfaces incompatibilities.

End state:
- Gradle 9.5
- AGP 8.13
- compileSdk 35, targetSdk 34
- NDK r25c
- Kotlin 2.3 (if 11.6 landed without issue)

This combination is supported by Google and gives ~80% of the upgrade's
benefit at ~20% of the risk.

## What survives any revert

- The 15 JNI symbol names (step 07's guard).
- All test code (step 10).
- The Slint headless tests (unchanged by step 11).
- The release pipeline (step 09).
- On-device behaviour for existing installs (no manifest changes affect
  installed instances).

## Forward fix vs. revert

For most issues, forward fix is preferred over revert. A typical
sequence:

1. PR 11.4 lands.
2. A regression appears in capture pipeline lifecycle.
3. Bisect inside 11.4: which AGP 9 change triggered it?
4. Apply a forward fix (e.g., add `tools:replace` to manifest, or pin
   ProGuard rule).
5. CI green; resume rollout.

Revert is the right call only when:

- The forward fix requires upstream changes (e.g., GStreamer 1.30
  release).
- The fix would take > 1 week.
- The bug affects production users (rare during step 11; tests should
  catch it).

## Atomic revert PR

To revert all of step 11 in one shot:

```bash
git checkout main
git checkout -b devin/$(date +%s)-revert-step-11
git revert <sha-11.7>^..<sha-11.1>
git push -u origin devin/$(date +%s)-revert-step-11
gh pr create --title "Revert: refactor step 11 (build-stack upgrade)" \
    --body "Restoring pre-step-11 build stack."
```

CI may take longer than usual (re-downloads of old toolchains); the
revert is otherwise mechanical.

## Worst-case: keystore lost during AGP 9 rotation

If you accidentally generate a *new* keystore during the upgrade (e.g.,
because AGP 9's `signingConfigs` semantics differ enough that a fresh key
got minted), this is the **only irrecoverable** scenario:

- Existing Play Store installs cannot update to the new key.
- Mitigation: keystore should be provisioned via the GitHub secret
  (sub-step 09.9), so AGP 9 doesn't *touch* the keystore — only consumes
  it. Verify in 11.4's pre-merge checklist.

If you confirm the keystore is intact across the bump (`sha256sum
release.jks` is unchanged), this risk doesn't materialise.

## Lessons from past upgrades

If the team has done a similar bump recently:

- Document the actual time it took per sub-PR (target: < 2 days each).
- Document the manual smoke pass duration (target: < 30 min per device).
- Document any forward-fix PRs that landed after the bump.

This data fuels the next upgrade's planning (e.g., AGP 9 → 10 in 12
months).
