# 11.9 — PR 11.5: targetSdk 34 → 36 (Android 16)

The behaviour-changing PR. After this, the OS no longer applies Android
14 compatibility shims.

## Diff

```diff
 # app/build.gradle
 android {
-    compileSdk 35
+    compileSdk 36
     defaultConfig {
-        targetSdk 34
+        targetSdk 36
     }
 }
```

`compileSdk` bumps to 36 in the same PR; running with `targetSdk = 36`
on `compileSdk = 35` errors at build time.

## Behaviour changes from API 35 → 36

Most relevant to this project:

### Orientation / resizability (large screens)

Android 16 on foldables:

- `screenOrientation="portrait"` no longer locks foldable's unfolded
  state to portrait. Users see the app stretched.
- `resizeableActivity="false"` is ignored on large-screen devices.

Mitigation: change to `userPortrait` and `resizeableActivity="true"`:

```diff
 <activity
     android:name=".MainActivity"
-    android:screenOrientation="portrait"
-    android:resizeableActivity="false"
+    android:screenOrientation="userPortrait"
+    android:resizeableActivity="true"
     android:configChanges="orientation|screenSize|screenLayout|smallestScreenSize|keyboardHidden">
```

### Foreground service types (API 34+)

If step 01.8 (manifest hardening) hasn't landed yet, add explicit
foregroundServiceType:

```xml
<service
    android:name=".ScreenCaptureService"
    android:foregroundServiceType="mediaProjection"
    android:exported="false" />
<service
    android:name=".GstPopService"
    android:foregroundServiceType="dataSync"
    android:exported="false" />
<service
    android:name=".MigrationRuntimeService"
    android:foregroundServiceType="dataSync"
    android:exported="false" />
```

Without these, `startForeground` throws on API 34+.

### Permission requests

API 36 enforces some permissions more strictly:

```bash
rg -n 'uses-permission' app/src/main/AndroidManifest.xml
```

Audit each:

- `FOREGROUND_SERVICE_MEDIA_PROJECTION` (required for screen capture FG).
- `FOREGROUND_SERVICE_DATA_SYNC` (required for backend FG services).
- `POST_NOTIFICATIONS` (already required since API 33; double-check
  declared).

### Storage scoping

API 30+ enforces scoped storage. If the project still references
`Environment.getExternalStorageDirectory()`:

```bash
rg -n 'getExternalStorageDirectory\|MediaStore\|FileProvider' app/src/main/
```

…audit and migrate to `MediaStore` or `app-specific` paths. Most likely
not relevant here (the project doesn't read external storage), but
verify.

## Audit script

After applying the diff, run a manifest audit:

```bash
rg -n 'screenOrientation\|resizeableActivity\|configChanges\|foregroundServiceType' \
    app/src/main/AndroidManifest.xml
```

For each match, decide:

- Is the value still appropriate on Android 16?
- Does the activity need updated `configChanges`?
- Are foreground-service types declared with the granular flags
  required from API 34+?

## Manual test plan (foldable / large screen)

After the PR lands:

1. Install on a Pixel Fold or Samsung Galaxy Z Fold.
2. Launch the app.
3. Unfold the device. App should adapt smoothly, no stretching.
4. Start a cast. Verify capture works on the unfolded display.
5. Fold the device back. App should adapt.
6. Rotate. Test in both orientations.

If the foldable is not available, an Android Studio emulator with the
"7.6\" Foldable" profile is a passable substitute (less faithful but
catches most layout issues).

## Rollback

```bash
git revert <sha-of-11.5>
```

Returns to `targetSdk = 34`. The OS will start applying Android 14
compatibility shims again. Useful if a foldable regression appears.

## Why "Medium risk"

`targetSdk` bumps are intermediate-risk:

- Compile-time impact: zero.
- Runtime impact: a handful of API-specific behaviour changes.
- Mitigation: a thorough manifest audit catches most issues pre-merge.

The remaining risk is foldable-specific behaviour, which the team can
absorb on-device with a fix-forward PR.
