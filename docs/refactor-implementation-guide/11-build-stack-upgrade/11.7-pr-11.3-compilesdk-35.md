# 11.7 — PR 11.3: compileSdk 34 → 35

Bump `compileSdk` to 35 (Android 15) **without** changing `targetSdk`
(stays at 34). This unlocks API 35 symbols for compile-time checks
without changing runtime behaviour.

## Diff

```diff
 # app/build.gradle
 android {
-    compileSdk 34
+    compileSdk 35
 }
```

## What changes

- The compiler can resolve API 35 symbols (e.g., new
  `android.app.WallpaperManager` methods).
- New deprecation warnings appear for things deprecated in API 35.
- `androidx` libraries may now require `compileSdk ≥ 35` (e.g.,
  `androidx.core` 1.13+).

## What does NOT change

- Runtime behaviour. `targetSdk` still 34, so the OS applies Android 14
  compatibility shims.
- `minSdk` unchanged at 26.

## Audit for deprecations

```bash
./gradlew --no-daemon :app:assembleDebug 2>&1 | grep -E 'warning:' | head -20
```

Expect a small number of new warnings. Decide for each:

- **Fix now:** if it's a one-line change.
- **Track:** if it's a behavioural shift that needs design discussion.
- **Suppress:** `@SuppressWarnings("deprecation")` for cases the team
  explicitly wants to defer.

## Library compatibility

```bash
rg -n 'androidx\.' app/build.gradle | head -20
```

For each `androidx.*` dependency, verify it's compatible with
`compileSdk = 35`. Most are; check the changelogs of:

- `androidx.core:core-ktx`
- `androidx.activity:activity-ktx`
- `androidx.lifecycle:lifecycle-*`
- `androidx.appcompat:appcompat`

The standard fix is to bump the dep version. Don't bump everything at
once — one at a time, gated on green CI.

## SDK install in CI

The composite action installs the SDK platform on demand. With
`compileSdk = 35`, add to `.github/actions/android-ci-setup/action.yml`:

```diff
- sdkmanager "platforms;android-34"
+ sdkmanager "platforms;android-34" "platforms;android-35"
```

(Or omit the explicit list and let AGP request it via `sdkmanager`
automatically.)

## Time impact

| Phase             | Before (compileSdk 34) | After (compileSdk 35)   |
|-------------------|------------------------|--------------------------|
| `assembleDebug`   | ~95 s                   | ~95 s                    |

`compileSdk` doesn't affect build time materially; it changes the API
surface visible to the compiler.

## Pre-merge

- [ ] Local: `:app:assembleDebug` green.
- [ ] No new lint errors (warnings allowed).
- [ ] CI green.
- [ ] Real device smoke.

## Why "Low risk"

Bumping `compileSdk` without `targetSdk` is the safest possible upgrade
— the runtime sees no change. The worst case is a `@RequiresApi` lint
that surfaces an existing bug; that's a feature.
