# 05.5 — `AppGraph.kt`

The composition root. One instance, owned by `FcastApp`.

Create `app/src/main/java/org/fcast/android/sender/AppGraph.kt`:

```kotlin
package org.fcast.android.sender

import android.content.Context
import org.fcast.android.sender.capture.CaptureEngine
import org.fcast.android.sender.capture.ScreenCaptureCoordinator
import org.fcast.android.sender.data.AndroidSecretStore
import org.fcast.android.sender.data.SecretStore
import org.fcast.android.sender.runtime.JniRuntimeBridge
import org.fcast.android.sender.runtime.RuntimeBridge

/**
 * Production composition root.
 *
 * Wires every long-lived dependency the application needs. One instance per
 * process; constructed in FcastApp.onCreate. Activities and services read
 * dependencies from here rather than constructing them ad-hoc.
 *
 * Replaces:
 *   - Static calls to GstPopServiceBridge / MigrationRuntimeServiceBridge.
 *   - The process-global Rust BACKEND singleton (via JniRuntimeBridge).
 *
 * Tests use a different AppGraph that returns FakeRuntimeBridge etc.
 */
class AppGraph(
    private val appContext: Context,
) {
    val runtime: RuntimeBridge by lazy {
        JniRuntimeBridge(appContext)
    }

    /** The coordinator is created lazily so it doesn't bind GL state at process start. */
    fun newCaptureCoordinator(
        callbacks: ScreenCaptureCoordinator.CaptureCallbacks,
    ): ScreenCaptureCoordinator = ScreenCaptureCoordinator(
        applicationContext = appContext,
        callbacks = callbacks,
        engineFactory = { CaptureEngine() },
    )

    val secretStore: SecretStore by lazy {
        AndroidSecretStore(appContext)
    }

    companion object {
        /** Test-only constructor in step 10. Production code never calls this. */
        @JvmStatic
        internal fun forTest(
            runtime: RuntimeBridge,
            secretStore: SecretStore,
            captureCoordinatorFactory: (ScreenCaptureCoordinator.CaptureCallbacks) -> ScreenCaptureCoordinator,
        ): AppGraph = object : AppGraph(/* not used */ appContext = NoopContext) {
            override fun newCaptureCoordinator(callbacks: ScreenCaptureCoordinator.CaptureCallbacks) =
                captureCoordinatorFactory(callbacks)
            // …override accessors via separate Kotlin properties; simplified for guide…
        }
    }
}
```

> The `companion object` `forTest` is sketch code — Kotlin does not let an
> anonymous subclass directly override `val` declarations of a class. In
> practice the test path uses a `TestAppGraph` class inside `app/src/test/`
> rather than a companion factory. Treat the sketch as documentation of the
> intent, not as something to paste verbatim.

## Why a separate `AppGraph` class?

The Application object should be the smallest Android-platform thing
possible. Keeping a tiny `AppGraph` separate makes it:

- Easy to test (pass a fake graph to `MainActivity` in instrumentation tests).
- Easy to extend (add `val timeProvider: Clock` in step 12, etc.).
- Easy to reason about lifetime — the graph is alive iff the process is
  alive.

## Why lazy properties?

- `runtime` doesn't need to exist until the user actually triggers a backend
  call. Saves ~5 ms at cold start.
- `secretStore` triggers `MasterKey` generation on first read. The Android
  Keystore call can be expensive (10–50 ms); deferring it until needed is
  the right trade-off.
- `captureCoordinator` is **not** a `lazy` field because it depends on a
  caller-supplied callbacks parameter — it must be a factory method.

## Why does `newCaptureCoordinator` take `callbacks`?

The coordinator's callbacks point at `MainActivity` methods. The activity is
short-lived (rotation can recreate it), so the coordinator can't be a
process-level singleton. `AppGraph.newCaptureCoordinator(myCallbacks)` is
called once per activity creation; the coordinator's `shutdown()` is called
in `onDestroy`.

This is the *only* method on `AppGraph` that is not a `val`. Every other
field is lazy-singleton.
