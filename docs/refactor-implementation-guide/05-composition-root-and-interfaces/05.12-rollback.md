# 05.12 — Rollback

Three layers, three independent revert points.

## Layer 1 — Kotlin side only

If the Kotlin composition root (`FcastApp` + `AppGraph` + `RuntimeBridge` +
`JniRuntimeBridge`) is the cause of a regression:

```bash
git revert <sha-of-the-Kotlin-side-PR>
```

The Rust side is untouched. The activity goes back to calling
`GstPopServiceBridge.start(...)` directly. The Rust registry stays installed
but unused (production code still calls the deprecated `backend::current()`).

## Layer 2 — Rust side only

If the Rust registry / App context cause a regression:

```bash
git revert <sha-of-the-Rust-side-PR>
```

`src/app.rs` and `src/backend/registry.rs` go. The deprecated shims become
no-ops referencing the old globals — which means **you must also restore the
original `BACKEND: Lazy<RwLock<…>>` body** in the same revert. Verify the
revert restores it:

```bash
git diff main HEAD -- src/backend/mod.rs | head -40
```

The diff should show the legacy `static BACKEND` declaration restored.

The Kotlin side stays. `JniRuntimeBridge` continues to call
`GstPopServiceBridge` — those services still exist and still work because
they have their own `Java_..._GstPopServiceBridge_nativeStart*` symbols.

## Layer 3 — Slint wiring only

If the Slint callbacks (sub-step 05.10) regress, revert just the
`src/lib.rs` (or `src/application/mod.rs` after step 07) hunk that changes
each callback. Each callback is independent; you can revert one without
reverting the others.

```bash
# Show the per-callback hunks:
git log --oneline -p src/lib.rs | grep -A 5 'on_start_'
```

Pick the offending callback and selectively revert.

## What survives any revert

- The new Kotlin classes (`FcastApp`, `AppGraph`, etc.) are orphaned but
  harmless on disk after a Kotlin-side revert.
- The Rust `src/app.rs` and `src/backend/registry.rs` files are orphaned but
  harmless after a Rust-side revert.
- No on-device state, no manifest schema change (except `android:name=".FcastApp"` —
  revert this too if reverting the Kotlin side).

## Cleanup after a revert

Orphaned files left on disk should be removed in a follow-up commit so the
codebase doesn't accumulate dead code. Tag the follow-up PR with `cleanup-only`.

## Partial rollback example

Step 05 introduces deprecation warnings. If the warning volume is overwhelming,
keep the `#[deprecated]` attribute but downgrade it to a documentation
comment until the migration PR lands:

```diff
-#[deprecated(note = "Use crate::app::app().registry() — see refactor step 05.")]
+/// **Deprecated** — use `crate::app::app().registry()`. See refactor step 05.
 pub fn current() -> std::sync::Arc<dyn MediaBackend> { … }
```

This is a deliberate trade-off: less noise during the soak, less compile-
time enforcement of the migration. Re-enable the attribute once the
migration PR is in flight.
