# 05.6 ā€” `FcastApp.kt` + manifest entry

Create the `Application` subclass that owns the `AppGraph`.

## New file

Create `app/src/main/java/org/fcast/android/sender/FcastApp.kt`:

```kotlin
package org.fcast.android.sender

import android.app.Application

/**
 * Single Application subclass; owner of the production [AppGraph].
 *
 * Activities and services read dependencies via:
 *
 *     val graph = (applicationContext as FcastApp).graph
 *     graph.runtime.startEmbeddedBackend(...)
 *
 * Do NOT add any other state here; everything goes through AppGraph.
 */
class FcastApp : Application() {
    val graph: AppGraph by lazy { AppGraph(applicationContext) }
}
```

## Manifest change

```diff
diff --git a/app/src/main/AndroidManifest.xml b/app/src/main/AndroidManifest.xml
@@
 <application
+    android:name=".FcastApp"
     android:allowBackup="true"
     android:icon="@mipmap/ic_launcher"
     android:label="@string/app_name"
     android:supportsRtl="true"
     android:theme="@style/Theme.FCast">
```

This is the *only* manifest change in step 05.

## Usage from `MainActivity` (Java today; Kotlin in step 08)

```java
private AppGraph appGraph() {
    return ((FcastApp) getApplication()).getGraph();
}

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    coordinator = appGraph().newCaptureCoordinator(captureCallbacks);
    coordinator.attach();
    runtime = appGraph().getRuntime();
}
```

The Kotlin property `graph` is exposed to Java as `getGraph()` automatically.

## Usage from the existing `GstPopService` / `MigrationRuntimeService`

The services keep calling their *own* `nativeStart*` symbols ā€” they do not
read `AppGraph` because they are entry points to the existing JNI shims.
Routing them through `AppGraph` is **out of scope** for step 05; the
report's recommendation about "no UI callback directly calls a static
service bridge" is enforced at the *UI*-side callsites (Slintā†’Rustā†’
`AppGraph.runtime`), not at the service-internal callsites.

## Why not Hilt / Dagger?

The report explicitly recommends "manual constructor injection, no heavy DI
framework":

> "Use explicit composition roots ā€” small graph builders, not Dagger/Hilt ā€”
> until the code base demonstrably outgrows them. The savings on build
> times and onboarding cost are larger than the cost of one constructor
> call per feature."

`AppGraph` is that small graph builder. If the app later grows past, say,
20 long-lived dependencies, revisit Hilt.

## Re-entrance check

```bash
rg -n 'class .* extends Application|class .* : Application' app/src/main
```

Must return exactly one match: `FcastApp.kt`. If the codebase already had an
`Application` subclass, merge its body into `FcastApp` and remove the old
class. The manifest entry chooses the one that wins; do not leave a dead
class on disk.
