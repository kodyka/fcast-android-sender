# 10.10 — CI: add JVM unit-test job

Add a step to the existing debug pipeline (`android-debug-apk.yml` after
sub-step 09.5's rename).

## Diff

```diff
 jobs:
   build-debug:
     runs-on: ubuntu-24.04
     steps:
       - uses: actions/checkout@v4
       - name: Set up Android CI toolchain
         uses: ./.github/actions/android-ci-setup
         with:
           rust-target: aarch64-linux-android
       - name: Build GStreamer Android JNI glue
         run: bash ci/build-gstreamer-android-glue.sh
       - name: Build arm64 Rust Android library (debug)
         run: bash ci/build-rust-android-lib.sh debug

+      - name: Run :app:testDebugUnitTest
+        run: ./gradlew --no-daemon :app:testDebugUnitTest --no-build-cache
+      - name: Upload unit-test reports
+        if: always()
+        uses: actions/upload-artifact@v4
+        with:
+          name: jvm-test-reports
+          path: app/build/reports/tests/testDebugUnitTest/
+          retention-days: 7

       - name: Build debug APK
         run: ./gradlew --no-daemon :app:assembleDebug
```

## Why `--no-build-cache`?

The first JVM test run primes Robolectric's internal cache; the second
run loads it. Caching at the Gradle level conflicts with Robolectric's
own state — disable.

## Why upload reports `if: always()`?

Failing tests block PRs. The HTML report is the primary debug surface;
attaching it on success *and* failure lets developers grab it without
re-running CI.

## Branch protection update

Add `Run :app:testDebugUnitTest` to required status checks (Settings →
Branches → main → Edit rule).

## Caching

The `actions/cache@v4` block from sub-step 09.4 covers the `~/.gradle/`
cache implicitly (Gradle stores under `~/.gradle/caches/`). After the
first JVM-test run is cached, the second run should complete in ~30 s.

If JVM tests slow down to > 2 min consistently, the JVM/JDK toolchain
download is the culprit — add explicit caching for `~/.gradle/jdks`.

## Failed-test re-run UX

GitHub Actions logs become hard to navigate when a test fails. The
uploaded artifact is the cleanest path:

1. Click the failing PR's "Details" link.
2. Click the "jvm-test-reports" artifact.
3. Download → unzip → open `index.html`.
4. The failure details (stack trace, message) are right there.

## What about flake-detection?

Robolectric is mostly deterministic — flakes are rare. If they happen:

1. Re-run the workflow.
2. If consistently flaky, add `@Test(retries=3)` (requires the
   `flaky-retry` library) or, simpler, identify and fix the test.

Do **not** mark tests `@Ignore` to "fix CI". That's the path to a
useless test suite.

## Verify the addition works locally

```bash
./gradlew --no-daemon :app:testDebugUnitTest 2>&1 | tee /tmp/jvm-tests.log
grep -E 'BUILD SUCCESSFUL|BUILD FAILED' /tmp/jvm-tests.log
```

Local runs use the same Gradle invocation as CI; if it passes locally,
it should pass in CI (modulo environmental differences like JDK version).
