## Layer-to-test mapping

| Layer            | What runs                          | Tests                                                                                                     |
|------------------|------------------------------------|-----------------------------------------------------------------------------------------------------------|
| JVM unit         | `:app:testDebugUnitTest`           | `ScreenCaptureServiceTest`, `StatusParserTest`, `SenderControllerTest`.                                    |
| Instrumented     | `:app:connectedDebugAndroidTest`   | `RuntimeBridgeInstrumentedTest`, `ScreenCaptureServiceInstrumentedTest`.                                   |
| Slint headless   | `cargo test -p fcastsender --test ui_snapshots`  | Existing, unchanged.                                                                          |

## What each test exercises

### `ScreenCaptureServiceTest` (JVM)

- Null intent on `onStartCommand` → returns `START_NOT_STICKY` and does
  not crash.
- Unknown action → ignored, returns `START_NOT_STICKY`.
- Granted result with valid data → `startForeground` is invoked
  (verified via `ShadowService.isForegroundStarted()`).
- Cancelled result → `startForeground` is never invoked.
- Notification channel with the service's channel ID exists after
  `onCreate` (verified via `NotificationManager.getNotificationChannel`).

Step 01's hardening lives here. The service contract is the most
fragile surface in the project.

### `StatusParserTest` (JVM)

- Valid JSON with `state=running`.
- Valid JSON with `state=error, message=…`.
- Invalid JSON → mapped to `state=error`.

The parser is small (~20 LOC) but every backend status round-trips
through it. Tests are cheap insurance.

### `SenderControllerTest` (JVM)

- `startBackend(MIGRATION, …)` with `state=running` → emits
  `UiState.Connected(MIGRATION, …)`.
- `startBackend(GSTPOP, …)` with `state=error` → emits
  `UiState.Error(…)`.
- `stopBackend(…)` → emits `UiState.Disconnected`.

Requires step 08.2 (controller existence). Tests use
`kotlinx-coroutines-test`'s `runTest` + `StandardTestDispatcher`.

### `RuntimeBridgeInstrumentedTest` (instrumented)

- `bridge.backendStatus(MIGRATION)` returns a JSON object the parser
  understands.

Verifies that the real JNI library is loadable and the JSON contract
matches expectations. Smallest possible instrumented test.

### `ScreenCaptureServiceInstrumentedTest` (instrumented)

- `startService(intent.setAction("unknown"))` does not throw.

Verifies the service starts via `Context.startService` (not just via
Robolectric's stub).

## Why these specific tests?

Each is a **minimum**: it covers the behaviour that most depended on by
the rest of the code, with the smallest assertion surface. As the suite
matures, additional tests can be added without changing the layout.

The aim is a coverage *floor*, not a coverage *ceiling*: hit 100% of the
service-contract paths, leave UI-level testing to Slint headless.
