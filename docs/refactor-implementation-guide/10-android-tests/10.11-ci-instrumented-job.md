# 10.11 — CI: add instrumented test job

Instrumented tests require a running emulator. Use
[reactivecircus/android-emulator-runner](https://github.com/ReactiveCircus/android-emulator-runner)
which handles emulator boot, retry, and snapshotting.

## New workflow `.github/workflows/android-instrumented-tests.yml`

```yaml
name: Android Instrumented Tests

on:
  pull_request:
    paths:
      - 'app/src/**'
      - 'src/**'
      - 'crates/**'
      - 'ci/**'
      - 'Cargo.lock'
      - '.github/workflows/android-instrumented-tests.yml'
  schedule:
    - cron: '0 4 * * *'  # nightly at 04:00 UTC, ~empty US time

permissions:
  contents: read

jobs:
  android-instrumented:
    runs-on: ubuntu-24.04
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4

      - name: Set up Android CI toolchain
        uses: ./.github/actions/android-ci-setup
        with:
          rust-target: aarch64-linux-android

      - name: Build GStreamer Android JNI glue
        run: bash ci/build-gstreamer-android-glue.sh

      - name: Build arm64 Rust Android library (debug)
        run: bash ci/build-rust-android-lib.sh debug

      - name: Enable KVM
        run: |
          echo 'KERNEL=="kvm", GROUP="kvm", MODE="0666"' \
            | sudo tee /etc/udev/rules.d/99-kvm.rules
          sudo udevadm control --reload-rules
          sudo udevadm trigger --name-match=kvm

      - name: Run connected tests
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: 34
          target: google_apis
          arch: x86_64
          profile: pixel_5
          ram-size: 4096M
          disable-animations: true
          script: |
            ./gradlew --no-daemon :app:connectedDebugAndroidTest

      - name: Upload connected-test reports
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: android-instrumented-reports
          path: app/build/reports/androidTests/connected/
          retention-days: 7

      - name: Upload screenshots
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: emulator-screenshots
          path: /tmp/android-screenshots/
          retention-days: 7
```

## Why `runs-on: ubuntu-24.04` and not the bigger runner?

Free GitHub-hosted Ubuntu runners support KVM acceleration. The
`reactivecircus/android-emulator-runner` action handles all the setup;
no need for a custom large runner unless tests are slow (> 15 min).

If your org has 4-core or 8-core runners and the emulator is slow, switch
to `runs-on: large-arm64` or similar.

## Why API 34?

- It's the current minSdkVersion target in `app/build.gradle` after
  step 11 lands.
- It's the API where `foregroundServiceType` is mandatory for media
  projection — the test surface.
- `google_apis` (not `google_play_apis`) is the lighter image — saves
  download time.

## Why `disable-animations: true`?

Some instrumented tests time out waiting for animations to complete.
Disabling animations makes them deterministic and ~3× faster.

## Schedule strategy

- **PR trigger:** Run on every PR that touches `app/src/**` or `src/**`.
  This ensures regressions are caught at PR time, not the next day.
- **Nightly cron:** Run on `main` daily to catch regressions from merged
  PRs that didn't trigger the path filter.

If runner cost is a concern, gate the PR trigger on a `needs-emulator`
label:

```yaml
on:
  pull_request:
    types: [labeled, opened, synchronize]
    paths: [ … ]
jobs:
  android-instrumented:
    if: contains(github.event.pull_request.labels.*.name, 'needs-emulator')
```

Then the team labels security-/lifecycle-related PRs to opt-in.

## Caching

The emulator AVD itself is cacheable:

```yaml
- name: AVD cache
  uses: actions/cache@v4
  with:
    path: |
      ~/.android/avd/*
      ~/.android/adb*
    key: avd-${{ runner.os }}-android-34-google_apis-pixel_5
```

`reactivecircus/android-emulator-runner` checks this cache; on hit, the
emulator boots in ~30 s instead of ~3 min.

## Flakiness mitigation

```yaml
- name: Run connected tests
  uses: reactivecircus/android-emulator-runner@v2
  with:
    api-level: 34
    script: |
      for i in 1 2 3; do
        ./gradlew --no-daemon :app:connectedDebugAndroidTest && exit 0
        echo "Attempt $i failed; retrying"
      done
      exit 1
```

Three retries cover most emulator hiccups. If a test fails consistently
across all three, it's a real failure.

## Branch protection

`Android Instrumented Tests / android-instrumented` is a candidate for
required status check — but:

- Required → blocks merge if emulator flakes.
- Not required → PRs can merge without instrumented coverage.

The pragmatic middle ground: required on `main` (post-merge gate), not
required on PR (advisory). Most teams accept this trade-off.

## Local invocation

```bash
emulator -avd Pixel_5_API_34 -no-snapshot -no-audio &
adb wait-for-device
./gradlew --no-daemon :app:connectedDebugAndroidTest
```

`Pixel_5_API_34` is an AVD name; create it via:

```bash
sdkmanager --install "system-images;android-34;google_apis;x86_64"
avdmanager create avd -n Pixel_5_API_34 \
    -k "system-images;android-34;google_apis;x86_64" \
    -d "pixel_5"
```

## Pre-merge checklist

- [ ] Workflow file lands.
- [ ] At least one PR triggers the workflow successfully.
- [ ] Nightly cron triggers on `main`.
- [ ] AVD cache is hit on the second run.
- [ ] Branch-protection setting reflects the team's chosen policy.
