# 10.14 — Rollback

## Per file

Each test file is independently revertible:

```bash
git revert <sha-of-PR-that-added-the-file>
```

The Gradle dependencies persist; they don't break anything in their
absence. Optionally remove them too, but it's harmless to leave them
declared.

## Revert order

Doesn't matter — tests are additive. There is no inter-dependency
between the test files.

## Revert the CI workflow

Sub-step 10.10 added a JVM-test step inside `android-debug-apk.yml`:

```diff
- - name: Run :app:testDebugUnitTest
-   run: ./gradlew --no-daemon :app:testDebugUnitTest --no-build-cache
- - name: Upload unit-test reports
-   if: always()
-   uses: actions/upload-artifact@v4
-   with:
-     name: jvm-test-reports
-     path: app/build/reports/tests/testDebugUnitTest/
-     retention-days: 7
```

…and 10.11 added a separate workflow. Revert each as needed:

- Keep JVM tests, drop instrumented → revert only the new workflow.
- Drop all CI gates → revert both.
- Drop just one test class → leave CI, but delete the file.

## Revert the test-thread cleanup (10.12)

```diff
- run: cargo test backend::gstpop -- --include-ignored
+ run: cargo test backend::gstpop -- --include-ignored --test-threads=1
+ # --test-threads=1 required: two ignored tests share process-global atomics
```

The serial run is a known-good fallback. Use this if a parallel run
surfaces an intermittent failure that you can't immediately diagnose.

## What survives any revert

- **No on-device state.** Tests don't write files outside the test
  process.
- **No app behaviour change.** Tests observe; they don't modify
  production code paths.
- **The Slint headless tests** continue to gate PRs (they pre-date step
  10).
- **The 15 JNI symbols** (step 07's stability guard is unaffected).

## Revert vs. `@Ignore`

If a single test is flaky and you want to defer fixing it without
reverting the whole class:

```kotlin
@Ignore("Flaky on API 34 — TODO: fix before next release. See #N.")
@Test fun running_status_yields_connected() = runTest { … }
```

This keeps the test in version control as a TODO and removes it from the
green/red gate. **Use sparingly.** Every `@Ignore` is a documented
regression.

## Rollback the entire step (worst case)

If the team decides to drop all of step 10:

```bash
git revert <sha-of-10.14>
git revert <sha-of-10.13>
# …all the way back to 10.4…
git revert <sha-of-10.4>
git revert <sha-of-10.9>  # Gradle deps last (test files reference them)
```

Or, faster, branch from `main` and cherry-pick *out* the relevant commits.

Final state: pre-step-10. Zero JVM tests, zero instrumented tests, Slint
headless still in place.

## What you lose by reverting

- The signal that the service contract still works as designed.
- The signal that the JNI bridge is loadable.
- The fast (< 1 s) local feedback for parser logic.

These tests are the **first** Android tests in the project; reverting
them returns the project to "manual testing only" mode. Strong
recommendation: even if a single test is broken, fix it forward rather
than reverting the suite.
