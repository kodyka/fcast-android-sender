## How to verify zero ABI drift

Before each sub-PR:

```bash
cargo build --release \
  --target=aarch64-linux-android \
  -p fcastsender
nm -D --defined-only \
  target/aarch64-linux-android/release/libfcastsender.so \
  | grep Java_org_fcast_android_sender \
  | sort \
  > /tmp/symbols.before.txt
```

After the PR's commits:

```bash
cargo build --release --target=aarch64-linux-android -p fcastsender
nm -D --defined-only \
  target/aarch64-linux-android/release/libfcastsender.so \
  | grep Java_org_fcast_android_sender \
  | sort \
  > /tmp/symbols.after.txt
diff /tmp/symbols.before.txt /tmp/symbols.after.txt
```

The diff must be empty. If it isn't, the PR has accidentally renamed or
dropped a symbol — block the merge.

## Symbol-stability CI gate

Add a CI step (see step 09's CI consolidation) that runs the above `nm
| diff` against the previous commit's `.so` to enforce the no-drift rule
automatically. Until that CI lands, this manual check stays in the
pre-merge checklist.

## Why `extern "C"` re-exports in `lib.rs`?

`#[unsafe(no_mangle)]` on a function in `jni_bridge::main_activity` works
fine — but having the re-export wafer in `lib.rs` makes the full symbol
table grep-able from a single file. Reviewers can see, at a glance, what
JNI surface the crate exports. The cost is ~3 lines per symbol; acceptable.

If you prefer not to have the wafer, drop `#[unsafe(no_mangle)]` on the
`lib.rs` re-export and place it on the `jni_bridge::*` function directly.
Symbol behaviour is identical.
