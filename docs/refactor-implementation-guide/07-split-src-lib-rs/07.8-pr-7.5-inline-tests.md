# 07.8 — PR 7.5: relocate inline tests

`src/lib.rs:3124+` declares an inline `mod phase9_dispatch_tests`. Move it
next to the code it exercises.

## Locate

```bash
rg -n '^mod .*_tests' src/lib.rs
```

Verified hit (May 2026):

```
3124:#[cfg(test)] mod phase9_dispatch_tests { … }
```

The body exercises dispatch helpers that, after PR 7.1, live in
`jni_bridge::helpers`.

## Move

Add to `src/jni_bridge/helpers.rs`:

```rust
#[cfg(test)]
mod phase9_dispatch_tests {
    use super::*;

    // …move body verbatim from lib.rs…
}
```

…and remove the block from `lib.rs`.

## Other inline test modules

Audit:

```bash
rg -n '^#\[cfg\(test\)\] *$|^#\[cfg\(test\)\]\nmod ' src/lib.rs
rg -n '^#\[cfg\(test\)\]' src/
```

Any test module sitting inside `lib.rs` should follow the same recipe:
move next to the code it tests. Specifically:

| Test module                 | Belongs in                          |
|-----------------------------|--------------------------------------|
| `phase9_dispatch_tests`     | `jni_bridge/helpers.rs`              |
| any `…_status_tests`        | `application/status.rs`              |
| any `…_probe_tests`         | `command/probe.rs`                   |
| any `…_legacy_tests`        | `command/legacy_tests.rs`             |
| any `…_application_tests`   | `application/mod.rs`                  |

If none of those test modules exist on `main`, this PR is empty for them —
just move `phase9_dispatch_tests` and call it a day.

## Test path discovery

Once relocated, run:

```bash
cargo test -p fcastsender --no-run 2>&1 | rg 'phase9_dispatch_tests'
```

The compiled test binary path will print. Each function inside should run:

```bash
cargo test -p fcastsender phase9_dispatch_tests
```

## Why not just leave them in `lib.rs`?

Inline tests in a 3000-LOC `lib.rs` are not discoverable; they don't add
value because nobody knows they exist. Co-locating tests with code makes
them visible during code review and lets the testing CI (step 10) target
specific modules.
