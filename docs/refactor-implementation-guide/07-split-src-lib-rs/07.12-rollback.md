# 07.12 — Rollback

Seven sub-PRs; seven independent revert points.

## Per-PR revert

```bash
# Pick the SHA of the PR you want to revert.
git revert <sha-of-PR-7.X>
```

Because every PR is mechanical (file move + path patches), each is safely
revertible in isolation. The most fragile case is PR 7.6 (crate promotion) —
discussed below.

## PR 7.1 / 7.4 / 7.5 / 7.7 revert

Each touches a single concept (helpers, `android_main`, tests, `unsafe`).
Reverting one does not affect the others. Symbol-diff guard catches any
accidental ABI change introduced by the revert itself.

## PR 7.2 / 7.3 revert

Reverting PR 7.2 alone (JNI body move) leaves PR 7.1's helper move in
place; the `lib.rs` JNI bodies after the revert reference
`jni_bridge::helpers::*`. That's fine — `jni_bridge::helpers` still exists.

Reverting PR 7.3 alone (Application carve-up) leaves the `application/`
module skeleton from PR 7.1; restored `Application` lives in `lib.rs`. Watch
for double-defined `default_presets` / `default_quick_actions` (both
`lib.rs` and `application/defaults.rs`): delete the duplicates from
`application/defaults.rs` as part of the revert.

## PR 7.6 revert (crate promotion)

This is the only sub-PR that changes the workspace graph. Reverting it:

```bash
git revert <sha-of-PR-7.6>
```

Restores the single-crate layout: every `crates/runtime-core` / `crates/
android-jni` module moves back under `src/`. The workspace edit unwinds in
the same commit.

Things to verify after the revert:

- `Cargo.toml` no longer mentions `runtime-core` / `android-jni` as
  dependencies.
- `workspace.members` no longer lists those crates.
- `cargo build -p fcastsender` green.
- The orphan `crates/runtime-core/` and `crates/android-jni/` directories
  are deleted (they ship in the same revert).

## What survives any revert

- No on-device state, no manifest schema change.
- The 15 JNI symbol names are preserved (the symbol-diff guard enforces
  this at CI time on the revert PR too).
- All test names are preserved (use `--list` diff to verify).

## Partial rollback example

Suppose PR 7.3 was merged, then later a regression appears that bisects to
PR 7.3's move of `Application::wire_slint_callbacks`. Two options:

1. **Hot-fix** the callback in-place. The post-PR-7.3 file location
   (`application/mod.rs`) is fixable directly; the bug is most likely a
   transcription error in the move.

2. **Selective revert.** Open a PR that does:

   ```diff
   -// in src/application/mod.rs
   -impl Application {
   -    pub fn wire_slint_callbacks(&self) { … }
   -}
   +// in src/lib.rs
   +// pasted-back version from the original lib.rs (pre-PR-7.3)
   ```

   The rest of PR 7.3 stays — only the offending method goes back.

Both options preserve the rest of step 07's progress.

## Atomic-revert exit door

If the entire step 07 needs to be undone (e.g., the team decides to halt
the refactor), revert PRs 7.1–7.7 in reverse order:

```bash
git revert <sha-7.7>
git revert <sha-7.6>
git revert <sha-7.5>
git revert <sha-7.4>
git revert <sha-7.3>
git revert <sha-7.2>
git revert <sha-7.1>
```

Resolve any noisy conflicts as they appear (mostly in `lib.rs`). Final state:
back to `main`'s pre-step-07 layout, `wc -l src/lib.rs` = 3076.
