# 07.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings":

> "`src/lib.rs` is 3,153 lines / 2,903 LOC and also acts as a major
> integration hub. […] In both places, multiple architectural layers are
> collapsed into single files: platform lifecycle, service control, media
> capture, rendering, JNI boundary, UI state, and backend orchestration."

Proposed mitigation (from "Refactor plan"):

> "**Extract Android JNI helpers, session orchestration, discovery/device
> connection, banner/state-handling, and backend switching into separate
> Rust modules/crates.** Keep the `lib.rs` to <300 LOC: it should only
> contain crate-level re-exports and the `Java_…` symbols."

## Verified line count (May 2026)

```bash
wc -l src/lib.rs
```

Result: **3076 lines**. The report's 3,153 is slightly stale (a recent PR
added a few hundred LOC, then dropped them). Treat 3076 as the working
target.

## Why "High effort"

This is the single largest refactor in the plan. It is broken into seven
PRs precisely because no reviewer can usefully evaluate a 3000-line move
in one go. Each sub-PR is < 500 LOC of net diff and independently shippable.

## Why "High priority" (not Highest)

The code works today. Splitting `lib.rs` is the prerequisite for steps 09–12
(CI improvements depend on per-crate test isolation; performance work
depends on benchmarkable units). It is not a *correctness* fix.

## Constraints

- **Every `Java_…` symbol stays on the same ABI position.** No symbol may
  be renamed, dropped, or moved into a sub-crate that doesn't end up in
  `libfcastsender.so`. Sub-step 07.3 contains the inventory.
- **No behavioural changes in 7.1–7.6.** These are file moves. Step 07.7
  is the only sub-PR that may alter `unsafe` code; even there it only adds
  `// SAFETY:` comments and replaces `from_raw` reconstructions with
  `OnceCell` borrows where the audit reveals a smell.
