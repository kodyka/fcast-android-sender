# 01.1 — Report finding

Direct quote from `deep-research-report-3.md`, "Detailed findings and issue
register":

> "`ScreenCaptureService` returns `START_STICKY`, but its `onStartCommand()`
> dereferences `intent` without a null guard; a sticky restart with a null
> intent would be unsafe. The same service uses `LocalBroadcastManager`, which
> AndroidX explicitly deprecated because it behaves like an application-wide
> event bus and encourages layer violations."

And the proposed mitigation (highest priority in the report's table):

> "**Harden `ScreenCaptureService`** — Make `onStartCommand()` null-safe,
> validate action/extras, return `START_NOT_STICKY`, and stop using a sticky
> contract for a one-shot permission hand-off."

The `LocalBroadcastManager` removal is its own step (**02**). This step (**01**)
focuses on the sticky / null-deref / action-validation contract.

## Why "highest priority"

The bug class is reachable from a routine Android cold-start path: process
death → service auto-restart with `null` intent → `intent.getIntExtra(...)` →
`NullPointerException` in a service that is supposed to be the bedrock of
screen capture. The fix is small (≤ 40 LOC), localised, and risk-isolated. No
later step depends on its design.
