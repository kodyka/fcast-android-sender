# 01.7 — Rollback

## Single-line revert

```bash
git revert <sha-of-the-merged-PR>
```

The PR touches two files:

- `app/src/main/java/org/fcast/android/sender/ScreenCaptureService.java`
- `app/src/main/java/org/fcast/android/sender/MainActivity.java` (one line)

The revert is byte-clean — no merge conflicts unless step 02 has landed in
between (in which case the LBM line will conflict; resolve manually).

## What survives the revert

- Nothing. There is no on-device state, no migration, no manifest change, no
  schema. Reverting takes the codebase back to byte-for-byte `main`.

## Partial rollback path

If the regression is *only* on cold-start after process death, restore the
`START_STICKY` line but keep the null guard:

```diff
-    return START_NOT_STICKY;
+    return START_STICKY;
```

That preserves the original auto-restart behaviour while removing the NPE
risk. Do not keep this hybrid form for long — file a follow-up to investigate
the regression.

## How to detect the regression

The two visible regressions are:

| Regression                                                         | Where it shows up                                              |
|--------------------------------------------------------------------|----------------------------------------------------------------|
| Capture fails to start on first permission grant                   | UI stays in "Disconnected" state; logcat shows
  `Missing screen-capture result payload` despite the consent dialog being accepted.       |
| Capture does not auto-start after the system kills the process     | UI starts fresh; expected on `START_NOT_STICKY`.               |

If the second case is the only regression, that is **not** a bug — it is the
explicit design of the patch. Restore via the partial rollback only if
product wants the auto-restart back.
