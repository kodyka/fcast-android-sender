## B1 — Recording page: disabled `Stop` button is still rendered

**Severity:** high
**Screenshot:** Recording page (`Snapshot_…16-50-26.png`) — small faded red rectangle below `00:00:00`.
**File:** <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/ui/pages/recording_page.slint" lines="159-169" />

The current code wraps the `Stop` `DestructiveButton` in a `HorizontalLayout` and toggles its `enabled` flag — but never gates *visibility*. With `enabled: false`, `DestructiveButton` paints at `opacity: 0.45` (see <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/ui/components/buttons.slint" lines="55-75" />), which leaves a faded red bar where there should be nothing.

```diff
                         // Stop button — only enabled when recording or paused.
-                        HorizontalLayout {
-                            alignment: center;
-                            spacing: Theme.spacing-default;
-                            DestructiveButton {
-                                label: @tr("Stop");
-                                enabled: root.mock-state == RecordingState.recording
-                                    || root.mock-state == RecordingState.paused;
-                                clicked => { root.on-stop-clicked(); }
-                            }
+                        if root.mock-state == RecordingState.recording
+                         || root.mock-state == RecordingState.paused: HorizontalLayout {
+                            alignment: center;
+                            spacing: Theme.spacing-default;
+                            DestructiveButton {
+                                label: @tr("Stop");
+                                clicked => { root.on-stop-clicked(); }
+                            }
                         }
```

**Why this is correct (per Slint docs).** `if <expr>: <element>` is a *conditional element* — the element is fully removed from the tree when the predicate is false, so the layout reclaims its space. Toggling `enabled` keeps the element in the tree at half-opacity, which is the visible bug.
**Doc:** `draft/slint-ui/docs/astro/src/content/docs/guide/language/coding/repetition-and-data-models.mdx` (the same conditional-element form is already used in `main.slint` — see <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/ui/main.slint" lines="71-76" />).

## B3 — Recording page: `idle` and `recording` use the **same** colour

**Severity:** high
**Screenshot:** Recording page (idle) — looks already-recording at a glance.
**File:** Same lines as B2.

## B7 — Bitrate presets: `Add preset` button looks like a list row

**Severity:** medium
**Screenshot:** Bitrate presets (`Snapshot_…16-52-12.png`).
**File:** <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/ui/pages/bitrate_presets_page.slint" lines="79-122" />

`PrimaryButton` is laid out inside the same `VerticalLayout` as the preset rows, with the same `Theme.spacing-default`. Because `PrimaryButton`'s height is `Theme.row-height` (48 px — same as a preset card) and its background is `Theme.accent` (similar steelblue tone to the active-row blue), the eye reads it as another preset. Add a separator and visually demote it:

```diff
                 for preset[i] in root.mock-presets: Rectangle {
                     /* … unchanged … */
                 }

+                // Visual separator between the list and the action.
+                Rectangle { height: 16px; }
+
                 PrimaryButton {
                     label: @tr("Add preset");
                     clicked => { Bridge.active-panel = Panel.bitrate-preset-edit; }
                 }
```

If you want the action to recede further, swap `PrimaryButton` → `TextButton` (and add a `+` glyph prefix: `label: "+ " + @tr("Add preset");`). The preset rows already carry the only "active" emphasis on this page; the add-action does not need to compete.

## B12 — Quick action bar: ignores Rust's `Bridge.quick-actions` model

**Severity:** architectural — fixed by Phase 8
**Screenshot:** Connect page (bottom bar in `Snapshot_…16-50-04.png` / `…16-50-13.png`) — shows 8 actions.
**Files:**
- <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/ui/components/control_bar.slint" lines="45-58" /> (page-local stub)
- <ref_snippet file="/home/ubuntu/repos/fcast/senders/android/src/lib.rs" lines="988-1002" /> (Rust setter — currently writes to a property nobody reads)

**Diagnosis.** `lib.rs:1002` calls `ui.global::<Bridge>().set_quick_actions(model.into())` with **5** actions in debug builds (`scan-qr` + 4 server-test actions) and **1** action in release builds. This is exactly the **Cluster C2** migration in [`PHASE-8-bridge-migration-plan.md`](./PHASE-8-bridge-migration-plan.md).

> launch, because `lib.rs:988-1000` currently only pushes that subset. Apply
> Part 1 (Slint) **and** Part 2 (Rust) in the same commit, or the regression
> is visible end-users.
